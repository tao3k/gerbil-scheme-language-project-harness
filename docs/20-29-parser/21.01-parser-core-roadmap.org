* 21.01 Parser Core Roadmap
:PROPERTIES:
:JD_ID: 21.01
:END:

** 21.01.1 Goal

The Gerbil Scheme harness parser is the source of truth for all provider facts.
Search, query, check, JSON packets, and future schema parity must consume facts
owned by the native Gerbil reader/expander boundary, not ad hoc text scans.

** 21.01.2 Parser Fact Layers

1. Native read layer.
   Preserve the exact boundary between module metadata returned by
   ~core-read-module~ and body syntax forms returned by the reader.
2. Form fact layer.
   Record every top-level body form with head, kind, source range, and selector.
3. Module fact layer.
   Keep module id, package, prelude, namespace, imports, exports, and includes as
   first-class file facts.
4. Definition fact layer.
   Derive definitions from syntax datums while preserving source locations and
   selectors.
5. Call fact layer.
   Derive native application facts from definition and form bodies with callee,
   arity, source path, and selector.
6. Expansion fact layer.
   Add phase-aware imports/exports and macro-expanded facts only behind commands
   that explicitly need expanded semantics.

** 21.01.3 Parity Targets

- Match the other language harnesses on compact search, owner-local query,
  direct-source-read, check, guide, and provider metadata.
- Keep language-neutral packet fields aligned with shared schemas.
- Put Gerbil-specific facts under provider-owned fields until the shared schema
  has an RFC-backed cross-language contract.
- Avoid declaring JSON support for a command until that command emits a schema
  validated packet.

** 21.01.4 Implementation Phases

*** 21.01.4.1 Phase 1: Parser Basis

- Export parser API accessors for source files, definitions, and top-level forms.
- Preserve top-level form facts in ~source-file~.
- Add parser tests for native module metadata, body form sequence, form kinds,
  and selectors.
- Validate against the harness fixtures and real Gerbil source.

*** 21.01.4.2 Phase 2: Parser Module Split

- Move parser data structures and native read functions out of ~cli.ss~ into a
  parser-owned module.
- Keep CLI rendering and command dispatch separate from parser facts.
- Add fixture tests that import the parser module directly.

*** 21.01.4.3 Phase 3: Schema-Oriented Packets

- Define a provider-owned Gerbil parser fact packet.
- Emit JSON only after the packet shape validates against local schemas.
- Add tests for search prime, owner query, direct-source-read, check, and doctor
  packet compatibility.

*** 21.01.4.4 Phase 4: Expander Facts

- Add commands that opt into Gerbil expander/module-context work.
- Capture phase-aware imports, exports, macro definitions, and module references.
- Keep side-effectful project reader or prelude execution out of default search.

*** 21.01.4.5 Phase 5: Real-Source Evidence

- Run parser and check passes against ~.data/gerbil~.
- Classify native reader findings into Gerbil modules, Gambit compatibility
  files, and custom grammar DSL files.
- Feed any shared-contract gaps back through RFC and schema changes before
  changing root ASP behavior.

** 21.01.5 Current Completed Step

The parser now preserves top-level body form facts with kind, head, path, source
range, and selector. Module headers returned as native metadata remain separate
from body forms, matching the Gerbil reader/expander boundary.

The parser-owned data structures, native read functions, source collection, and
project index now live behind ~src/parser/facade.ss~.
The CLI imports that module and keeps command dispatch/rendering separate from
parser fact ownership.

The parser also exports native ~call-fact~ values through ~source-file-calls~
and ~project-calls~. The first slice records ordinary application callee names
and arity from native syntax datums. Call facts now also preserve argument
datum strings, literal argument type names, and the enclosing definition name
when the caller is known, so type-owned checks can resolve argument types
through native parameter and literal facts. Selectors intentionally point at the
enclosing top-level form until nested syntax spans are promoted into the parser
model.
