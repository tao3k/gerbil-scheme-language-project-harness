* 21.02 Gerbil Harness Layout

** 21.02.1 Source Boundaries

The harness keeps native parser authority under the =parser/= namespace and
keeps protocol projection under the =protocol/= namespace. CLI modules may
coordinate commands, but they should not own parser facts or schema-shaped JSON
projection.

#+begin_src text
src/
  cli.ss
  constants.ss
  commands/
    agent.ss
    check.ss
    guide.ss
    query.ss
    search.ss
  parser.ss
  parser/
    core.ss
    query.ss
  protocol/
    json.ss
    registry.ss
  snapshot/
    core.ss

  support/
    args.ss
    io.ss

#+end_src

** 21.02.2 Parser Boundary

=src/parser/core.ss= owns native reader integration, source collection,
top-level form facts, definition facts, and project indexing.
=src/parser/facade.ss= is a stable facade for command modules and tests, so the
internal parser namespace can keep splitting without forcing broad import churn.

** 21.02.3 Protocol Boundary

=src/protocol/json.ss= owns JSON projection for parser-owned facts.
=src/protocol/registry.ss= owns registry projection. Search and query commands
can reuse these projections, but the parser core remains Scheme data first and
does not depend on JSON output.

=src/snapshot/core.ss= owns stable test projections for schema-facing facts.
Snapshot fixtures use =.ss= files because they are Gerbil-native Scheme data
read by tests, not generic =.scm= source modules or JSON golden files.

Package extension JSON stays provider-owned under the shared search packet
=extensions= field. Poo is currently reported with =dependencyMode=required=
because activation comes from a concrete =gerbil.pkg= =depend:= entry managed
by =gxpkg=. The extension facade exposes model facts first through
=project-extension-facts=; JSON and compact search lines are projections of the
same fact model. Poo packet stability is covered by
=test/snapshots/poo-extension-packet.ss=. Poo scenario coverage includes mixed
=gerbil.pkg= dependency lists so Agent edits preserve the full package context,
not only the extension activation token.

** 21.02.4 Test Boundary

Top-level =test/*-test.ss= files are test entrypoints. Fine-grained test logic
lives under =test/unit/= by domain, such as =test/unit/search/=,
=test/unit/schema/=, and =test/unit/snapshot/=. Unit modules export higher-level
Scheme check functions or fact projections; entrypoints own =test-case= labels
and compose those functions. This keeps the module grain closer to the Rust
harness unit layout while preserving Gerbil-native test logic.

** 21.02.5 Next Split

The next parser split should move project walking and package/config facts out
of =src/parser/core.ss= once parser-facts packets require richer root metadata.
