* What the POO Sources Get Right
** Domain sections are architectural boundaries
=io.ss= is the clearest compact example, but the same discipline appears
across the other POO sources.  =io.ss= is organized by protocol domain:

1. byte I/O
2. representation printing
3. JSON I/O
4. marshaling
5. byte conversion
6. string conversion
7. writeenv printing
8. typed values

Each section introduces one protocol layer and keeps related fallback
logic nearby.  =object.ss=, =table.ss=, =trie.ss=, and =type.ss= do the
same thing for runtime state, table capabilities, trie representation,
and typed descriptors.  These files are not small, but they are locally
understandable when read by protocol section.

** Generic declarations are contracts, not incidental calls
The =.defgeneric= forms are compact contract points.  Their comments
state direction and arity, and defaults stay adjacent to the generic.
This makes the default behavior visible without forcing callers to read
method implementations.

Examples of the pattern:

- =object-read-byte=
- =read-bytes-into=
- =object-read-bytes=
- =object-write-bytes=
- =marshal=
- =unmarshal=
- =bytes<-=
- =<-bytes=
- =string<-=
- =<-string=

** Methods extend protocols without taking ownership of the protocol
The =:pr=, =:json=, =:write-json=, and =:wr= methods are not generic
condition dumps.  They are protocol extensions with explicit fallback
order.  The method owns dispatch and fallback; adapter types own derived
capabilities.

** Adapter types compose capabilities
The =define-type= adapters are the strongest reusable design signal:

#+begin_src scheme
(define-type (methods.string<-json @ [] .json<- .<-json)
  .string<-: (compose string<-json .json<-)
  .<-string: (compose .<-json json<-string))
#+end_src

This is the shape the harness should reward:

- small derived capability type
- explicit required slots
- no hidden string heuristic
- no duplicated conversion logic in every consumer

** Local helpers stay local
In =:pr= and =:wr=, helpers such as =d=, =s=, =w=, =comma=, =sn=, and
=wr-fields= are scoped inside the method or immediately adjacent to it.
They reduce visual weight without exporting accidental abstractions.

** Representation invariants are adjacent to the data they explain
=trie.ss= documents branch height, skip height, costep key reconstruction,
and unstep callback contracts directly next to the structs.  This is a
different kind of quality from smallness: dense data-structure code is
acceptable when invariants are local, typed, and test-backed.

** Protocol adapters translate roles explicitly
=Set<-Table.= and =RationalSet= show the adapter style the harness should
reward.  They do not duplicate table operations.  They translate table
key/value callbacks into set element callbacks and keep the dependency
adapter boundary visible.

* Current Harness Gap
The harness has the right semantic direction, but the source and tests
are still heavier than this quality bar.

** POO parser classifications now have data-owned tables
=src/parser/poo.ss= now has better names such as =poo-form-slots= and
=poo-form-options=.  The next code pass also moved the main option and
slot classifications into explicit parser-owned constants:

- generic options: =slot:=, =from:=, =default:=, =compute-default:=
- object slot syntax: =?=, ===>=, ===>.+=, =(next-method ...)=
- define-type options: =sealed:=, =transparent:=, =slots:=,
  =constructor:=, =final:=

This matches =io.ss=, where protocol names and fallback order are visible
as first-class structure rather than being rediscovered in branch bodies.
The remaining parser work is not to add more hard-coded branches; it is
to grow these small tables when new parser-owned POO syntax evidence
requires it.

** POO structural tests are still assertion-heavy
=t/unit/search/structural-index.ss= now has
=+poo-structural-fixture-owners+= and =check-structural-index-poo-facts=,
which is better than mixing every POO assertion into the generic
structural test.

The remaining problem is that =check-structural-index-poo-facts= is still
a long flat checklist.  It should become data-driven:

#+begin_src scheme
(poo-fact-spec

          (receiverType . "<Line>")
          (dispatchArity . 3))

The test should iterate facts, not duplicate the assertion shape for
every fixture.

** Parser tests repeat the same fixture ceremony
The POO parser tests repeatedly perform the same sequence:

1. parse a fixture
2. collect =source-file-poo-forms=
3. find one form by name
4. check role, supers, slots, options, and specializers

This should be collapsed into helpers such as:

- =find-required-poo-form=
- =check-poo-form-core=
- =check-poo-form-slots-contain=
- =check-poo-form-query-shape=

This is not a cosmetic cleanup.  Without it, every new high-quality
fixture makes the tests more fragile and harder for agents to edit.

* 2026-06-17 Policy Rule Transfer
The next policy transfer from this source study is now anchored in the
build-support and package-build agent rules.

The relevant upstream evidence is:
- =gerbil-poo/proto.ss= models extension as small prototype combinators:
  =identity-proto=, =compose-proto=, =compose-proto*=, and
  =instantiate-protos=.
- =gerbil-poo/io.ss= models I/O behavior as protocol boundaries:
  =.defgeneric= contracts, =defmethod= extensions, and adapter methods
  carry ownership instead of spreading string checks through consumers.
- =gerbil-utils/cli.ss= keeps CLI input handling as a small helper surface
  rather than shell-template orchestration.

Harness policy should therefore keep R019/R020 on their current split:
- R019 owns top-level =build.ss= responsibility drift, such as package build
  CLI routing and wrapper-definition ownership.
- R020 owns build-runtime quality drift through composite detections.
- R020 build-support shell-template drift requires more than helper names. A
  helper-name group by itself is evidence, not a warning.
- R020 can warn only when independent parser-owned groups agree, such as
  helper definitions plus shell-control literals or shell writer calls.
- R020 package-build shell-pipeline drift remains stricter: =sh -c= dispatch
  and pipeline payload evidence must both be present.
- Positive runtime wrappers must stay quiet for both R019 and R020 when they
  use Gerbil process APIs and typed argv lists.

The regression tests now encode these constraints:
- =agent policy requires multiple build-support evidence groups=
- =agent policy accepts Gerbil runtime build-support wrappers=

This is the shape we want for future rules: source-backed evidence groups
come from the native parser, POO-style detector prototypes combine them, and
the policy warning is emitted only after the composite gate is satisfied.

The scope guard for this rule family has also moved one step closer to that
architecture. =src/parser/source-class.ss= now classifies =build.ss= as
=package-build= and =build-support/*.ss= as =build-support-runtime=. R019 and
R020 consume those parser-owned classes instead of each policy file redoing
path string tests. The new parser regression
=source path class owns build policy scope= protects that boundary.

This turns the fixture suite into a POO best-practices corpus rather than
a pile of parser examples.

The native parser now exposes =projectionBurstFacts= for caller-scoped emitter/projection shape. The fact is intentionally permissive: it records a caller that mixes field projections such as =hash-get= with output calls such as =displayln=. It carries field keys, access count, distinct field count, emitter count, emitters, selectors, and quality facets. The parser does not decide that the code is bad; it only gives the agent a stable native fact.

R016 consumes that fact through a POO/C3 detection prototype named =emitter-projection-burst-all-of=. The warning requires three independent evidence groups:
- =high-projection-access-count=
- =multi-field-projection=
- =emitter-output-boundary=

This prevents the previous hard-coded shape mistake. A single high =hash-get= count is only evidence. A single output call is only evidence. A warning appears only when access density, key spread, and output boundary all align.

The source witness is =gerbil-utils-projection-builder=, backed by =gerbil-utils/base.ss= composition helpers plus Gerbil list-builder/fold examples. These source witnesses now use the same =slot-profile= C3 machinery as dependency adapter profiles, so detection details can expose both =detectionProfilePrecedence= and =detectionSourceProfilePrecedence=. The repair instruction therefore stays open: the agent may extract selectors, line builders, folds, or local helpers, but it must keep traversal and formatting as separately testable behavior.

Regression evidence:
- =native reader captures projection-burst parser facts=
- =agent policy reports emitter projection burst through multiple signals=
- =source detection overlays expose POO profile precedence=

This is the desired pattern for future quality rules: native parser facts first, source-backed POO profile overlays second, multi-signal combinator gates third, and model-guided repair last.

* Working Definition of "Best Practice"
For this harness, "POO best practice" means:

1. the source is organized around protocol boundaries;
2. generic declarations expose contracts;
3. methods own dispatch and fallback, not data conversion policy;
4. adapter types compose derived capabilities;
5. parser facts expose those boundaries to agents;
6. production policy/profile code uses C3 slot profiles for override order when
   multiple independent concerns compose;
7. tests validate the fact contract through data, not repeated hand-coded
   assertion blocks.

That is the quality bar demonstrated by =.data/gerbil-poo/io.ss=,
=object.ss=, =mop.ss=, =table.ss=, =trie.ss=, =type.ss=, and
=rationaldict.ss=.  The Gerbil harness should converge toward that bar
through parser facts, guide output, and data-driven tests before adding
new warning rules.
