** 11.12.1 Research Question

Gerbil describes itself as providing "a state of the art macro and module
system on top of the Gambit runtime and compiler".  The same upstream note says
that the macro system is based on =quote-syntax=, has a full meta-syntactic
tower, implements =syntax-case= natively, and uses single-instantiation modules
to support ahead-of-time compilation and compiled macros.

For the harness this cannot remain a slogan.  The parser, policy, scenario, and
agent guidance need a concrete model of what those words mean in source.

*** 11.12.1.1 Working Mental Model

Read Gerbil's macro/module system as one pipeline with four layers:

1. Syntax object layer
   - The reader/expander does not treat source only as raw Scheme lists.
   - Identifiers can carry source, lexical context, and marks.
   - =quote-syntax= preserves that context; =syntax->datum= deliberately drops
     it; =datum->syntax= deliberately reattaches it from a template.

2. Native macro-pattern layer
   - =syntax-case= matches syntax objects, not strings.
   - Literal identifiers are compared by binding/context, not by printed name
     alone.
   - Ellipses, pattern variables, fenders, and splices are explicit expansion
     facts.

3. Phase and module-context layer
   - =current-expander-phi= records whether code is runtime code, syntax-time
     code, or a higher macro phase.
   - A =module-context= owns namespace, path, imports, exports, delayed syntax
     evaluation, and expanded code.
   - Imports and exports carry phase, so compile-time and runtime dependencies
     are distinct facts.

4. Compiler split layer
   - The compiler consumes expanded module-context code.
   - Runtime code, meta/syntax code, phase code, and optional SSXI optimizer
     metadata are generated from the same core AST.
   - Compiled macros work because syntax-time code is emitted as a compiled
     artifact that can be loaded during later expansion.

This is the key point: Gerbil macros, modules, and AOT compilation are not
separate features glued together.  The module system gives macros a stable
phase-aware home, and the compiler preserves that home by emitting both runtime
and meta artifacts.

*** 11.12.1.2 Why This Is Easy To Misread

Basic Scheme mental model:

#+begin_example
source text -> lists -> eval
#+end_example

Gerbil's model for real modules:

#+begin_example
source text
  -> syntax objects with context/marks
  -> phase-aware macro expansion
  -> module-context with import/export facts
  -> expanded core AST
  -> runtime artifact + meta/syntax artifact + optimizer metadata
  -> Gambit compilation
#+end_example

An agent that only knows the first model writes broad list walkers, datum-only
macros, ad hoc import scans, and duplicate module collectors.  The harness must
teach the second model through parser facts, diagnostics, and scenarios.

** 11.12.3 Macro System Model

Gerbil's macro system starts from syntax objects, not raw lists.  The upstream
expander defines an ordinary =AST= layer plus three syntax wrappers:
=identifier-wrap= holds an identifier with marks, =syntax-wrap= applies a
mark, and =syntax-quote= carries context plus marks.  This is the concrete
meaning of "based on quote-syntax": a quoted syntax object can preserve lexical
context and mark history instead of collapsing immediately to a datum.

=syntax-e= is therefore intentionally not just =car= or =cdr= over lists.  It
unwraps syntax while preserving marks on substructure.  =syntax->datum= removes
syntax context, while =datum->syntax= reattaches context and marks from a
template identifier.  Good Gerbil macros are written at this boundary: use
datum conversion only when the repair is intentionally leaving the syntax
world, and keep syntax context when generating identifiers or pattern-bound
terms.

The native =syntax-case= implementation is not a library-level string matcher.
It parses patterns, recognizes literal identifiers with
=bound-identifier=?=, records pattern variables with depth, builds matching
code, handles fenders, and generates splice loops for ellipses.  The generated
macro code itself is built from core forms such as =lambda%=, =let-values=,
=let-syntax=, =if=, and =quote-syntax=.  This explains why macro quality in the
harness cannot be judged by surface syntax alone: the important facts are
syntax context, pattern variables, ellipsis depth, fender placement, and phase.

** 11.12.4 Meta-Syntactic Tower

The tower is represented by =current-expander-phi= and =phi-context=.  Normal
runtime code starts at phase 0.  =begin-syntax= and =define-syntax= advance the
phase while compiling or evaluating syntax code.  User macro expanders remember
their creation context and phase, and bindings store their phase.

This makes macros compiled Gerbil code that runs in a phase-specific expander
world.  A macro can import modules, bind syntax, define helpers, and generate
runtime code while still carrying syntax context through =quote-syntax=.  The
harness should model this as phase-aware ownership, not as a flat top-level
definition list.

Harness facts that should come from this model:

- syntax owner: =defsyntax=, =defrules=, =define-syntax=, =let-syntax=, and
  =letrec-syntax= should be first-class owners.
- phase edge: =begin-syntax= and syntax definitions should create a compile-time
  edge distinct from runtime imports.
- syntax object operation: =quote-syntax=, =syntax=, =syntax-case=,
  =datum->syntax=, =syntax->datum=, and =syntax-e= should be indexed as macro
  semantics, not generic function calls.
- hygiene boundary: identifier generation should preserve context unless the
  source explicitly drops to datum.
- ellipsis/fender boundary: complex =syntax-case= macros need scenario facts for
  pattern depth, fenders, and splice loops.

** 11.12.5 Module System Model

Gerbil modules are represented by =module-context= objects.  A module context
has a namespace, path, imports, exports, delayed evaluation object, and quoted
expanded code.  Importing a source module performs a transaction:

1. reset expander context, marks, phase, and import/export phase parameters
2. read package/prelude/namespace/module body
3. build or import the prelude context
4. create a =module-context=
5. expand the module body in that context
6. quote the expanded body as a =%#begin= core AST
7. store delayed =eval-syntax*= in =module-context-e=
8. store the quoted expanded code in =module-context-code=
9. cache the context in =__module-registry= by path and module id

This is the concrete single-instantiation mechanism: the registry is consulted
before a module is imported again, and a module context is reused instead of
re-expanded and re-instantiated.  Submodules are also syntax-bound module
contexts.  Imports and exports are not string lists; they are =module-import=,
=module-export=, =import-set=, and =export-set= values with source context,
identifier key, name, phase, and weak-import information.

The harness implication is direct: a Gerbil module graph should be recovered
from module contexts and phase-adjusted import/export sets where possible.  A
plain source scan of =(import ...)= is useful for a fast seed, but it is not the
complete module system.

** 11.12.6 Ahead-Of-Time And Compiled Macro Boundary

The compiler consumes =module-context-code= after expansion.  It does not
compile arbitrary source text directly.  The driver imports a module through
the expander, then =compile-top-module= collects bindings, generates runtime
code, generates meta code, and optionally generates SSXI optimizer metadata.

Runtime code and meta code are separate compiler methods over the same core AST.
=%#define-values= contributes runtime definitions; =%#define-syntax= and
=%#begin-syntax= contribute meta/phase code; =%#quote-syntax= has dedicated
runtime generation that serializes marks and references module contexts.  This
is why compiled macros can work with AOT compilation: the compiler emits the
runtime part and the phase/meta part needed by future expansion.

For harness policy this means "compiled macro" is not an aesthetic rule.  It is
a boundary where the source creates reusable compile-time code.  Quality checks
should reward thin macro surfaces that preserve phase, isolate runtime helpers,
and avoid unnecessary datum conversion.  They should not force every abstraction
into a macro, and they should not rewrite measured runtime hot paths into
compile-time machinery.

** 11.12.7 Gambit Boundary

Gerbil sits above Gambit.  The expander and module system are Gerbil-owned, and
the compiler eventually generates Gambit Scheme/C artifacts through =gsc=.  The
Gambit boundary is therefore the runtime/compiler substrate, not the semantic
owner of Gerbil modules or hygienic macros.

This distinction matters for the harness:

- Gerbil macro/module facts should be extracted from Gerbil expander/compiler
  concepts first.
- Gambit primitives are relevant for performance and low-level runtime behavior,
  but they should not erase phase, context, or module facts.
- Build and benchmark policy should measure the real Gerbil path: source ->
  module context -> expanded core AST -> runtime/meta generation -> Gambit
  compilation.

** 11.12.8 Harness Research Gaps

Current harness research has good application-level policy notes, but the
source-root model is still incomplete.  The next improvement plan should add
scenario-backed parser and policy facts for:

- phase-aware owner extraction for =begin-syntax=, =define-syntax=,
  =defsyntax=, =defrules=, =let-syntax=, and =letrec-syntax=
- module-context import/export facts with phase and weak-import metadata
- syntax-object operation facts for =quote-syntax=, =datum->syntax=,
  =syntax->datum=, and =syntax-e=
- =syntax-case= pattern complexity facts: literal ids, fenders, ellipsis depth,
  splice loops, and pattern-variable depth
- compiled macro boundaries: runtime helper, syntax helper, meta section, and
  emitted phase code
- single-instantiation checks that detect repeated ad hoc module loading or
  bespoke source collectors that bypass the module registry
- performance scenarios that compare direct module-context extraction with
  broad source rescans

** 11.12.9 Policy Direction

The policy objective is to help agents write better Gerbil, not to hard-code
Gerbil source style into downstream packages.  Good policy should fire when
parser facts prove that the code is already at a macro/module boundary and the
repair can preserve Gerbil's native semantics.

High-value warnings should say:

- use syntax objects instead of datum-only generation when hygiene matters
- keep macro transformers thin and push runtime behavior into ordinary helpers
- use =syntax-case= when pattern depth, fenders, or ellipses are real
- keep compile-time imports and runtime imports distinct
- let module contexts own package/prelude/namespace/import/export facts
- do not rescan a package when the expander has already materialized the module
  context

Low-value warnings to avoid:

- "use macros" without a phase or syntax-object reason
- "use syntax-case" for simple =defrule= cases
- "use advanced syntax" without benchmark evidence
- broad style rewrites that hide runtime hot paths or module boundaries

** 11.12.10 Immediate Next Scenarios

The next scenario set should be complex enough to represent real agent-written
Gerbil, not toy syntax:

1. =macro-hygiene-context-preservation=
   - input: macro converts syntax to datum too early and generates identifiers
     without template context
   - expected: repair uses =syntax-case= plus =datum->syntax= at the local
     identifier boundary
   - benchmark: parse/check stays subsecond on repeated files

2. =phase-aware-helper-split=
   - input: one macro owner mixes =begin-syntax= helper code, runtime helper
     code, and generated runtime forms
   - expected: compile-time parser helpers and runtime helpers are split while
     keeping the public macro thin
   - benchmark: no duplicate full-project collection

3. =module-context-import-export=
   - input: ad hoc collector scans source imports/exports and misses prelude,
     namespace, and phase facts
   - expected: repair uses the module-context API boundary
   - benchmark: warm path reuses module context facts

4. =single-instantiation-build-boundary=
   - input: build/test helper repeatedly imports or evaluates the same module
     outside the registry path
   - expected: repair uses the normal module import path and cached context
   - benchmark: repeated test target does not re-expand the same module graph

These scenarios should be added before expanding policy further.  Otherwise the
harness will keep producing surface-level "advanced syntax" rules instead of
teaching agents the real Gerbil macro and module system.

** 11.12.11 Scheme Optimization Refactor Plan

*** 11.12.11.1 Decision And Current Baseline

The refactor treats Gerbil macro optimization as a parser and module-graph
problem, not as a larger collection of style rules. =src/policy/agent-style-gerbil-signals.ss=
is a governance consumer over parser/package facts; it is not the owner of
syntax-object, phase, or module-context semantics. It must remain a consumer
after the refactor.

The current scenario baseline is asymmetric:

- =macro-hygiene-context-preservation= has a scenario owner.
- =phase-aware-helper-split= and =module-context-import-export= do not yet have
  parser-owned scenario identities.
- =single-instantiation-build-boundary= is only a weak performance-policy
  association, not an executable scenario contract.
- no distinct macro/module performance-gate owner currently proves cold versus
  warm module-context behavior.

The first implementation goal is therefore to make the missing semantic facts
and scenarios real before widening policy coverage.

*** 11.12.11.2 Target Topology

#+begin_src mermaid
flowchart LR
  Source["Gerbil source"] --> Syntax["syntax-object facts"]
  Syntax --> Phase["phase-aware macro facts"]
  Phase --> Module["module-context graph"]
  Module --> Compiled["compiled macro boundary"]
  Compiled --> Facts["macro/module fact facade"]
  Facts --> Policy["checker and policy consumers"]
  Facts --> Scenarios["gxtest scenarios and receipts"]
  Scenarios --> Gates["latency and heap gates"]
  Gates --> Facts
#+end_src

The proposed parser-owned module family is:

- =src/parser/macro-model.ss=: language-neutral fact records and identities.
- =src/parser/macro-syntax.ss=: =quote-syntax=, datum conversion, hygiene, and
  =syntax-case= complexity facts.
- =src/parser/macro-phase.ss=: syntax owners, =begin-syntax=, phase edges, and
  compile-time versus runtime helper boundaries.
- =src/parser/macro-module-context.ss=: namespace, prelude, import/export,
  weak-import, and cached module-context facts.
- =src/parser/macro-compiled.ss=: runtime/meta/phase output and compiled-macro
  boundary facts.
- =src/parser/macro-facade.ss=: the only parser API consumed by checker, policy,
  scenario, and search layers.

=checker/forms= and =policy/*= consume this facade. They must not add another
source collector, list walker, phase classifier, or module cache.

*** 11.12.11.3 Refactor Stages

1. Establish the fact contract.
   Define parser-owned identities, phase-edge records, syntax-operation records,
   module-context records, compiled-boundary records, and receipt shape. Keep
   source and compiled evidence distinct. This stage is Scheme-native and does
   not add a JSON schema or search-packet field. A versioned provider schema is
   only considered later if a stable cross-provider search contract is actually
   required. No policy behavior changes in this stage.

2. Extract macro syntax and phase facts.
   Move macro-head recognition from generic form handling into the macro parser
   family. Model hygiene-sensitive datum conversion, literal identifiers,
   fenders, ellipsis depth, and syntax helper/runtime helper separation.

3. Build the module-context graph.
   Replace ad hoc import/export collection with module-context facts. Preserve
   prelude, namespace, import phase, export phase, and weak-import state. Cache
   by the native module identity so repeated analysis does not rescan the same
   module graph.

4. Connect compiled macro boundaries.
   Relate expanded source owners to runtime, syntax-time, meta, phase, and SSXI
   output facts. The compiler path may consume these facts, but this project
   must not reimplement the Gerbil expander or compiler.

5. Rewire consumers and delete duplicated logic.
   Make =checker/forms=, macro policies, search receipts, and scenario helpers
   consume =macro-facade=. Remove datum-only duplicate scanners, package-wide
   rescans, and policy-private module collectors once equivalent facts exist.

6. Optimize only after fact and scenario convergence.
   Use the module graph to remove duplicate expansion and collection work,
   reduce allocation in warm paths, and isolate compiler-heavy owners. Do not
   use a universal heap limit, an external RSS poller, or a new product CLI as
   an optimization substitute.

*** 11.12.11.4 Scenario And Gate Plan

Each scenario is a normal =gxtest= module. A resource-sensitive scenario opts
in with its own literal profile:

#+begin_src scheme
(declare-gxtest-memory-exception
 '((maxHeapMiB . <scenario-budget>)))
#+end_src

=gxi= enforces the declared managed-heap ceiling. A heap overflow exits the
child nonzero and the outer =gxtest= assertion fails. No scenario polls =ps= or
uses an outer timeout wrapper. Scenario construction stays on the existing
testing/build-support macro surface; it does not create a benchmark or build
subcommand.

Required scenario sequence:

1. Upgrade =macro-hygiene-context-preservation= to assert parser facts for the
   syntax-to-datum boundary and context-preserving identifier repair.
2. Add =phase-aware-helper-split= to assert separate syntax and runtime helper
   ownership with no duplicate full-project collection.
3. Add =module-context-import-export= to assert phase-aware import/export,
   namespace, prelude, and warm-context reuse.
4. Add =single-instantiation-build-boundary= to assert one native module graph
   import/expansion per repeated target.
5. Add a compiled-macro boundary scenario only after the first four facts are
   queryable through =macro-facade=.

Every scenario records cold and warm elapsed time with explicit =ms= units,
selected module count, repeated-expansion count, and its declared managed-heap
ceiling. Establish budgets from baseline data first; the only existing research
constraint is subsecond repeated hygiene parse/check behavior. A performance
gate fails only against the scenario's own declared budget.

*** 11.12.11.5 Migration And Deletion Rules

- No compatibility facade for datum-only macro semantics survives after the
  parser facts are available.
- No policy may infer phase or module ownership from a printed identifier or
  path substring when a parser fact exists.
- No build/test helper may import/evaluate a module outside the native registry
  path merely to obtain metadata.
- No global memory threshold is introduced; each exceptional scenario owns its
  own literal =maxHeapMiB= configuration.
- No =gslph= benchmark or build passthrough command is added. Build stays on
  the upstream =gxi= / =gxtest= foundation.

*** 11.12.11.6 Implementation Order And Exit Criteria

Implement one semantic slice at a time in this order: syntax facts, phase facts,
module-context graph, compiled boundary, consumer migration, then optimization.
For every slice, the exit criterion is a parser receipt, one focused scenario,
cold/warm latency evidence in =ms=, and a declared heap profile where allocation
risk exists. Only after the scenario passes may the older collector or policy
branch be deleted. The refactor is complete when all macro policy findings and
search receipts originate from =macro-facade= and repeated targets reuse one
module-context graph without a project-wide rescan.

*** 11.12.11.7 High-Return Use Of Macros: Compile Policy Metadata, Not Work

The highest-return macro application is not another syntax convenience. The
current policy entry point manually appends 42 rule families, while policy
owners contain about 60 =project-index-files= traversals. The parser must know
which fact families every enabled rule requires before it starts parsing if it
is to avoid constructing the complete rich =SourceFile= for every owner.

Introduce one compile-time =defpolicy-rule= form whose expansion registers:

- stable rule id and output order;
- execution stage: =file-local= or =project-finalizer=;
- required fact families;
- file applicability predicate;
- runner procedure;
- optional compact project-summary reducer.

The macro expands to ordinary typed descriptors and a static registry. It must
not parse source, walk directories, inspect the runtime module registry, or
perform policy work during expansion. Its value is that the runtime can compute
the union of required facts once, build a bounded parse plan, dispatch all
file-local rules while an owner is resident, and run only the declared
cross-file finalizers after the stream closes.

This makes the macro/module research operational: compile-time metadata removes
manual registry drift and enables demand-driven runtime work, while Rust remains
the owner of workspace indexing, owner enumeration, and global search graphs.
The macro slice is accepted only after a handwritten two-rule prototype proves
the streaming contract and its memory gate; metaprogramming is not the first
implementation step.
