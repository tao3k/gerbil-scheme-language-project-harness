* Summary

The Gerbil Scheme harness is on the right architecture path, but it is not yet
the best possible version of itself.

The strongest part is the boundary design: language capability is now expressed
as parser-owned facts, policy rules, guide exemplars, schema/snapshot evidence,
and bench/check closure.  That is materially better than telling an agent
"this is Scheme" and hoping it remembers syntax.  The harness can already expose
macro facts, binding facts, POO facts, higher-order facts, control-flow facts,
dependency usage facts, and package policy through machine surfaces.

The remaining gap is that the harness still has several places where the
engineering posture is implicit or duplicated.  It can detect and steer important
Gerbil capabilities, but it does not yet score whether a project is using those
capabilities well across production owners versus fixtures, does not yet catch
all search-vocabulary drift, and still duplicates some rule catalog material
across policy, guide, info, and snapshots.

Current assessment:

| Area | Status | Assessment |
|------+--------+------------|
| Parser-owned facts | Strong | Structural feed reports 119 owners, 1066 symbols, 17730 syntax facts, and 376 dependency usages with =rawSourceStored=false=. |
| Agent policy | Strong foundation | =check --full .= passes with findings=0 and =info.agentSteering= exposes R008/R009/R011/R012. |
| Gerbil-specific steering | Good but incomplete | POO, macro witness, higher-order, runtime-source, and package-module guidance exist; usage-quality scoring is still shallow. |
| Code elegance enforcement | Partial | The harness can warn about manual loops and vague shapes, but it does not yet produce an architectural "capability posture" score. |
| Macro governance | Correct direction | Runtime-source witness policy exists; expansion-level witness coverage and version vocabulary can be improved. |
| POO/protocol adoption | Good evidence surface | =defprotocol=, =defgeneric=, =defmethod=, class/interface facts, and POO extension facts are projected; design-opportunity detection should be deeper. |
| Performance closure | Good baseline | Bench passes at 2141 ms total; =structural-index-packet= is the slowest stage at 1388 ms and should become the next optimization target. |

* Answer: Is This Already The Best?

No.  It is strong, but not finished.

The current implementation has moved beyond basic syntax memory.  It already
gives an agent passive, machine-readable steering about Gerbil-specific
engineering choices.  However, "best" would mean:

1. The harness can quantify whether a project uses Gerbil capability as an
   engineering boundary, not just whether facts exist.
2. Policy, guide, info, search, and snapshots share one rule catalog instead of
   duplicating rule IDs and wording.
3. Search vocabulary is complete enough that agents can ask natural engineering
   questions and hit the right structural facts.
4. The harness can distinguish production use, fixture use, and test-only
   witness use for macros, POO, protocols, higher-order forms, and runtime
   evidence.
5. Bench and structural indexing have owner-level hot path diagnostics, not only
   a total stage duration.

The present state is a good P3/P4 platform.  The next step is a P5 engineering
quality loop that turns these surfaces into project posture scoring and
agent-facing repair paths.

* What Is Already Strong

** Package and module engineering

The harness understands Gerbil as a package/module system rather than a pile of
files.  The prime and info receipts expose =gerbil.pkg=, package manager, source
scope, build fallback, dependency usage, and configurable policy fields.  This is
the right direction for downstream projects whose sources may not live only
under =src/=.

Strength:

- Package owner is parser-visible.
- =gerbil.pkg= policy owns source scope and agent-policy overrides.
- =build.ss= can contribute runtime roots when source scope is not explicit.
- The info packet exposes closure commands for =check=, =bench=, and
  =selfApply=.

Gap:

- Search coverage for the terms =configurable= and =interface= missed in the
  package/configuration frontier even though the info packet has that material.
  Agents should be able to query "configurable interface" and land directly on
  =info.agentSteering= and =gerbil.pkg= policy facts.

** Parser-owned structural facts

The structural feed is the core success.  The harness now exposes native syntax
facts instead of asking agents to infer Gerbil dialect features from source text.

Strength:

- POO facts include class, generic, method, and protocol/interface surfaces.
- Higher-order facts include =lambda=, =map=, =cut=, and related functional
  idioms.
- Macro facts include =defsyntax= and =defrules= surfaces.
- Dependency usages are present and tied to =gerbil.pkg=.
- The feed explicitly says =rawSourceStored=false=.

Gap:

- The compact structural view is strong for seed evidence, but the harness needs
  a separate audit/profile view that summarizes capability posture:
  production owners using macro/POO/higher-order/runtime facts, test-only
  witnesses, and unexercised dependency capabilities.

** Policy surfaces that can steer agents

The important agent rules already exist:

- R008: POO method shape requires =defgeneric= plus class or protocol receiver
  evidence.
- R009: pure data transform prefers =map/filter/fold/for/fold/cut= while keeping
  named-let for IO/state/generator/continuation drivers.
- R011: macro edits require runtime-source witness evidence.
- R012: protocol-shaped owners require declared protocol evidence.

This is the right level: policy talks about engineering boundaries, not style
preferences.

Gap:

- The rule catalog is distributed across =src/policy/model.ss=,
  =src/policy/agent.ss=, =src/commands/info.ss=, =src/commands/guide.ss=, and
  snapshots.  That creates drift risk.  There should be one native Gerbil rule
  catalog that renders to policy, info, guide, search, and snapshots.

** Guide exemplars

The guide command has source-backed exemplars for higher-order control,
POO policy, and macro runtime-source witness.  That matters because an agent
learns from project-local idioms instead of generic Scheme snippets.

Gap:

- The current guide frontier is small.  It should expose more progressive
  exemplars:
  - protocol implementation repair
  - manual object encoding to POO migration
  - macro edit with runtime-source selector and expansion test
  - named-let preservation for stateful drivers
  - functional transform rewrite for pure collection flow

** Runtime witness boundary

The runtime-source and compare surfaces are one of Gerbil's real advantages over
small Scheme scripts.  The harness already has runtime-source snapshots and
policy hooks for version-sensitive macro/runtime behavior.

Gap:

- The runtime frontier missed natural =gxi= and =version= query terms.  Agents
  often ask "what gxi version is active?" or "is this 0.18.2 behavior?".  The
  compact projection should include active =gxi --version=, Gerbil source ref,
  and tag/commit facts as searchable evidence.

* Code Quality Audit

** Architecture and ownership

The module layout is generally healthy.  There are clear owners for parser,
policy, commands, protocol, snapshots, and tests.  =check --full .= passes with
zero findings, which means the current policy suite does not see facade bloat,
legacy test layout, vague names, or known generated-code policy drift.

Main concern:

- Passing policy does not prove architectural elegance.  It proves that the
  existing rule set did not fire.  The harness needs "opportunity" checks that
  identify capability underuse and duplication even when there is no violation.

Recommended improvement:

- Add a =capability posture= report with statuses:
  - =absent=: no parser-owned facts for this capability.
  - =test-only=: present only in fixtures/snapshots/tests.
  - =production-evidence=: present in =src/= or package runtime owners.
  - =policy-covered=: policy can warn or repair.
  - =guide-covered=: guide can show source-backed idiom.
  - =snapshot-covered=: behavior has stable snapshot evidence.
  - =bench-covered=: performance is measured.

** Macro use and governance

Macro governance is better than a ban list.  The harness knows =defsyntax=,
=defrules=, and =syntax-case= as facts and routes macro edits through
runtime-source witness policy.

Main concern:

- Macro edit safety is currently expressed as witness policy, but the report
  surface should make the exact witness requirement obvious:
  active runtime version, runtime-source selector, macro owner, and expansion or
  roundtrip test.

Recommended improvement:

- Add =macroWitnessCoverage= facts:
  - macro owner
  - macro kind
  - active runtime source selector
  - expansion/roundtrip test owner
  - policy override state
  - missing witness fields

** POO, protocol, and object modeling

The POO/protocol path is meaningful.  The harness has =defprotocol= parsing,
method/generic shape policy, POO extension facts, and guide content that names
=defclass=, =defgeneric=, =defmethod=, and =defprotocol=.

Main concern:

- The current rule catches incorrect method/protocol shape, but it should also
  identify design opportunities.  If a project declares or imports POO and then
  repeatedly models objects as ad hoc alists/hashes, the harness should not wait
  for a broken =defmethod= to warn.  It should suggest a POO boundary.

Recommended improvement:

- Add =manual-object-encoding= facts from parser call/binding facts:
  - repeated associative keys
  - hash/table object-like access
  - loosely paired constructor/accessor functions
  - dependency context indicating POO is available
  - suggested =defclass= / =defprotocol= / =defgeneric= repair path

** Higher-order and control-flow idioms

The higher-order parser and policy are a major improvement over R5RS-style
memory.  The harness recognizes =lambda=, =case-lambda=, sequence operations,
partial application, and manual loop contexts.

Main concern:

- The policy can warn about manual loops, but it needs confidence and
  preservation reasons.  Some named-let forms are the right Gerbil shape for
  stateful IO/generator/continuation drivers; others should become
  =map/filter/fold/cut=.

Recommended improvement:

- Add =functionalTransformCandidate= facts:
  - purity confidence
  - accumulator shape
  - stateful call presence
  - continuation/generator role
  - suggested replacement idiom
  - named-let preservation reason when not rewriting

** Rule catalog and renderer duplication

The agent rule IDs appear in multiple owners and snapshots.  This is useful for
rendering, but dangerous as the rule set grows.

Recommended improvement:

- Create a native Gerbil =rule-catalog= owner that defines each rule once:
  - id
  - topic
  - severity
  - evidence kind
  - policy predicate
  - guide topic
  - compact renderer text
  - schema/export shape
  - tests/snapshots

Then make =policy=, =info=, =guide=, and compact search derive from that catalog.

** Search projection quality

The search frontiers found the right owners, but several broad engineering
queries came back with =queryQuality=low= and =package-drift=.  That is expected
for long natural-language audit queries, but it points to a product gap.

Observed missing or weak search vocabulary:

- =configurable= and =interface= missed in package/configuration search.
- =gxi= and =version= missed in runtime witness search.
- =defrules= was not fully covered in one macro governance query even though
  =defrules= facts exist.
- =engineering=, =features=, =redundancy=, =elegance=, and =steering= were weak
  in the broad audit query.

Recommended improvement:

- Add an =audit= or =capability= search profile.  It should fan out internally
  to package, policy, structural facts, runtime-source, guide, tests, snapshots,
  and bench, then return a compact synthesis rather than forcing the agent to
  issue one broad lexical query.

** Performance

The current bench is passing, but =structural-index-packet= dominates runtime.
This makes sense: the structural feed is doing the most useful work.  It is also
the next optimization target.

Recommended improvement:

- Add per-owner timing buckets to structural indexing:
  - parse/read time
  - structural fact extraction time
  - dependency usage time
  - JSON/render time
  - file hash/cache time

- Add an incremental bench:
  - warm unchanged project
  - one changed parser file
  - one changed policy file
  - one changed test fixture
  - deleted file refresh

* Policy Gaps To Add

** P0: Rule catalog drift

Finding shape:

- id: =GERBIL-SCHEME-ENG-R001=
- trigger: same rule id appears in policy/info/guide/snapshot but fields differ
- repair: derive all renderer surfaces from one rule catalog owner

Why it matters:

- Agent steering must be stable.  A mismatch between policy and guide means the
  agent can receive different repair instructions from different commands.

** P0: Search vocabulary coverage

Finding shape:

- trigger: documented/info/guide terms are not searchable through compact
  =search fzf= or owner queries
- repair: add query keys to structural/search projection

Initial terms:

- =configurable=
- =interface=
- =gxi=
- =version=
- =defrules=
- =engineering=
- =capability posture=
- =agent steering=

** P0: Production capability posture

Finding shape:

- id: =GERBIL-SCHEME-ENG-R003=
- trigger: capability is available only through fixtures/tests or guide examples
  and not used or witnessed in production owners
- repair: either add production witness, document why test-only is sufficient,
  or lower the capability claim

Why it matters:

- It prevents the harness from claiming "we use Gerbil's high-level features"
  when only fixtures demonstrate them.

** P1: Manual object encoding while POO is available

Finding shape:

- id: =GERBIL-SCHEME-AGENT-R013=
- trigger: repeated alist/hash object patterns under a package that declares POO
  capability
- repair: suggest =defclass=, =defprotocol=, =defgeneric=, or =defmethod=
  depending on shape

** P1: Macro edit lacks expansion witness

Finding shape:

- id: =GERBIL-SCHEME-AGENT-R014=
- trigger: macro owner changed or generated macro introduced without expansion
  test, roundtrip test, or runtime-source selector
- repair: run runtime-source query, add expansion witness, then edit

** P1: Higher-order rewrite confidence

Finding shape:

- id: =GERBIL-SCHEME-AGENT-R015=
- trigger: manual loop over pure data without state/IO/continuation evidence
- repair: suggest =map=, =filter=, =fold=, =for/fold=, or =cut=
- suppression: named-let preservation role with explicit state/IO/generator
  reason

** P2: Structural-index hot owner

Finding shape:

- id: =GERBIL-SCHEME-PERF-R001=
- trigger: one owner or fact family dominates structural index runtime
- repair: add incremental cache, split owner, or reduce repeated extraction

* Agent Steering Target

The agent should receive this kind of passive guidance from the harness:

- This is a Gerbil package module.  Preserve package, namespace, import/export,
  and =gerbil.pkg= policy style.
- POO capability is available.  Prefer =defclass=, =defprotocol=,
  =defgeneric=, and =defmethod= when the code is modeling object/protocol
  behavior.
- Macro edits require a runtime-source witness and an expansion or roundtrip
  test.  Do not edit transformer shape from memory.
- Pure data transforms should prefer =map/filter/fold/for/fold/cut=.  Preserve
  named-let when facts show IO/state/generator/continuation driver behavior.
- Runtime-sensitive behavior should route through active runtime witness and
  source acquisition before relying on remembered Gerbil behavior.

This should be generated from parser-owned facts and policy evidence, not from a
static prompt paragraph.

* Bottom Line

The harness is no longer just "Scheme syntax support".  It is already a
Gerbil-specific engineering harness with package facts, macro witness policy,
POO/protocol facts, higher-order/control-flow facts, runtime-source evidence,
snapshots, and bench/check closure.

The next quality jump is not adding more prose.  It is turning those facts into
a capability posture system that can answer:

- Is this project using Gerbil's advanced capabilities in production code?
- Are those capabilities covered by policy, tests, guide examples, and
  snapshots?
- Can the agent see the right repair path before it writes low-quality Scheme?
- Are expensive structural facts incremental and measurable?

That is the boundary where Gerbil becomes an engineering language for agents,
not just a more capable Scheme dialect.
