** 11.04.1 Scope

This audit extends =11.03= from "important examples" to a more complete map of
advanced Gerbil/Scheme features that should shape
=gerbil-scheme-language-project-harness= policy and guide output.

The important distinction is:

- =.data/gerbil= is the language implementation. Its source demonstrates
  engineering-level semantics: expander state, hygiene, phase, module loading,
  MOP, runtime control, actors, syntax errors, and pattern compilation.
- =.data/gerbil-utils= is an advanced library. Its source demonstrates dense
  language usage: autocurrying, combinators, call/cc generators, coroutines,
  syntax helpers, builder boundaries, and compact algebraic comments.

The harness should learn from both, but with different policy meanings. Runtime
implementation facts should become source-backed authority. Utils facts should
become style exemplars and guide samples.

** 11.04.3 Complete Feature Matrix

*** 11.04.3.1 Syntax objects, ASTs, marks, and syntax errors

Source: =.data/gerbil/src/gerbil/runtime/syntax.ss=.

Key features:

- AST values preserve datum plus source.
- Syntax errors carry message, irritants, where, context, phi, and marks.
- Macro matching uses syntax objects, generated identifiers, and hygienic
  identifier comparison.
- Error display intentionally projects source/context rather than a generic
  runtime stack trace.

Harness meaning:

- Macro policy must not be based on token strings alone.
- Comment guidance for macro owners should ask for hygiene/phase/boundary
  rationale when parser facts show syntax-object work.
- The evidence graph should retain source, context, and phase as first-class
  facts when available.

LLM blind spot:

- Models often treat Scheme macros as list rewriting. In Gerbil, macro code is
  a source/context/mark-aware AST program.

*** 11.04.3.2 Expander module system and phase state

Source: =.data/gerbil/src/gerbil/expander/module.ss= and
=.data/gerbil/src/gerbil/core/module-sugar.ss=.

Key features:

- Module import/export records include source, name/key, phi, and weak flags.
- Import/export expanders are user expanders with current expander context.
- Module loading tracks registries, package cache, prelude context, expander
  path, current marks, and current phi.
- =for-syntax=, =for-template=, =only-in=, =except-in=, =rename-in=,
  =rename-out=, and related forms are semantic DSLs, not decorations.

Harness meaning:

- Module edits need namespace/phase evidence, not only import text.
- Search packets should expose phase-bearing import/export transforms as
  structured facts.
- Guide examples should teach phase-aware module style from source-backed
  samples.

LLM blind spot:

- Models often flatten imports into runtime dependencies. Gerbil import/export
  forms can change expansion-time surfaces and should be reasoned about as
  module-transform DSLs.

*** 11.04.3.3 MOP and compile-time class/type generation

Source: =.data/gerbil/src/gerbil/core/mop.ss=.

Key features:

- MOP modules generate class type identifiers from the current expander
  context.
- =defclass-type= and =defstruct-type= are syntax-driven type generators.
- Class metadata includes id, name, super list, slots, precedence list,
  ordered slots, struct/final/system flags, metaclass, constructor, predicate,
  accessors, mutators, and unchecked accessors.
- Generated code carries MOP annotations for class, constructor, accessor,
  mutator, and predicate facts.

Harness meaning:

- POO/protocol policy should prefer parser-owned =defclass=, =defmethod=,
  =defgeneric=, and generated type facts over alist/hash approximations.
- Search should distinguish "object data encoding" from "MOP-owned class
  semantics".
- Guide repair should show class/type evidence when suggesting POO rewrites.

LLM blind spot:

- Models may see =defclass= as a runtime object convenience. In Gerbil it also
  participates in compile-time metadata, expansion annotations, and type layout.

*** 11.04.3.4 Extensible match DSL

Source: =.data/gerbil/src/gerbil/core/match.ss=.

Key features:

- =match-macro= extends =macro-object=.
- Pattern parsing happens at syntax time.
- Patterns cover gated tests, algebraic combinations, cons/list/vector/box,
  values, struct/class patterns, equality forms, quote/quasiquote,
  applicative destructuring, and user match macros.
- Variable extraction uses continuation-passing style and hygienic
  bound-identifier comparison.

Harness meaning:

- Repeated nested destructuring should trigger warning-level guidance toward
  match or helper extraction when parser facts show a stable shape.
- Policy must remain flexible because low-level parser/runtime code may need
  explicit conditionals.
- Guide examples should teach match as an extensible DSL, not merely as
  prettier =cond=.

LLM blind spot:

- Models often translate match to simple pattern syntax. Gerbil match can call
  syntax-local macro expansion and class/type metadata.

*** 11.04.3.5 Runtime control: promises, parameters, dynamic-wind, keywords

Source: =.data/gerbil/src/gerbil/runtime/control.ss=.

Key features:

- Atomic promises use mutexes, vector CAS, and =dynamic-wind= to protect
  single-entry forcing.
- =call-with-parameters= composes parameter rebinding recursively.
- =with-unwind-protect= expresses finalization through =dynamic-wind=.
- Keyword dispatch validates expected keywords, duplicate keywords, keyword
  escapes, and rest argument boundaries.

Harness meaning:

- =dynamic-wind=, parameterization, and keyword dispatch should be explicit
  evidence categories in control-flow policy.
- Code that uses continuation or cleanup boundaries should receive boundary
  comment guidance, not generic "prefer fold" style advice.
- Functional-transform policy must exempt resource/control owners.

LLM blind spot:

- Models may suggest replacing control loops with pure folds. That is wrong
  around =dynamic-wind=, parameters, keyword dispatch, or reentrant protection.

*** 11.04.3.6 Actors, continuations, thread locals, and diagnostic hooks

Source: =.data/gerbil/src/gerbil/runtime/thread.ss=.

Key features:

- Actor threads install exception handlers that capture continuations before
  unwinding.
- Debug hooks can inspect the continuation and exception before stack
  unwinding.
- Thread locals are implemented with actor-local tables, primordial-thread
  tables, weak-key hash tables, and locks.
- Lock helpers distinguish ordinary exception-safe locking from
  =dynamic-wind=-based locking.

Harness meaning:

- Actor/thread owners need policy that recognizes diagnostic hooks,
  continuation capture, exception unwinding, and thread-local state.
- Local mutation and locks should be evaluated under lifecycle/cleanup
  semantics, not blanket style warnings.
- Search evidence should expose actor/thread/control owner categories.

LLM blind spot:

- Models often discuss concurrency as only threads and locks. This source
  couples actor identity, continuation diagnostics, exception handlers, and
  thread-local storage.

*** 11.04.3.7 Autocurrying and altered application semantics

Source: =.data/gerbil-utils/autocurry.ss=.

Key features:

- =fn= and =defn= define semantically unary functions.
- Multiple arguments create closure chains until the final argument arrives.
- =%app= applies arguments in curried style.
- The file explicitly compares the semantics to ML and documents unsupported
  optional/keyword cases.

Harness meaning:

- Autocurry is a language-mode shift. Guide samples can use it as advanced
  material, but policy should not push it broadly into ordinary harness code.
- If a project imports autocurry, search should expose that application
  semantics differ from normal Scheme calls.

LLM blind spot:

- Models may silently assume Scheme varargs and normal procedure application.
  Autocurry deliberately changes the mental model.

*** 11.04.3.8 Combinators and algebraic comments

Source: =.data/gerbil-utils/base.ss=.

Key features:

- =rcompose= handles left-to-right multi-value function composition.
- =compose= wraps the conventional right-to-left order.
- =!>= and =!!>= encode pipeline flow for single and multiple values.
- =curry= and =rcurry= use =case-lambda= common-case specialization.
- =fold<-reduce-map= expresses a fold constructor from reduce and map.
- Comments use type-like transform signatures and only explain optimization
  boundaries when needed.

Harness meaning:

- This is the strongest exemplar for =typed-combinator-style=.
- Policy should require the three criteria at warning level for reusable
  exported helpers: contract shape, compact expression-level composition, and
  optimization-boundary comments.
- The policy must not require a single rigid text template; it should check for
  the presence and usefulness of the three roles.

LLM blind spot:

- Models often write explanatory prose above functions. Here the valuable
  comment is a compact algebraic contract that tells the transform shape.

*** 11.04.3.9 Generators, inversion of control, and coroutine fallback

Source: =.data/gerbil-utils/generator.ss=.

Key features:

- The file distinguishes iterators, iterating functions, and generating
  functions.
- Push and pull interfaces are explicitly treated as duals related by
  inversion of control.
- =generating<-for-each= uses =let/cc= to implement yield-style behavior.
- The source explains why this does not play well with finally-like blocks and
  provides a cothread/coroutine route.
- It records design alternatives: actor variants, monadic alternatives,
  delimited continuations, and SRFI 158 comparison.

Harness meaning:

- This is a guide exemplar for "do not blindly rewrite control flow".
- Search evidence should identify generator/coroutine/continuation owners as
  control-flow owners where normal functional-transform advice is weaker.
- Warnings should ask for lifecycle and finalization comments around
  continuation inversion.

LLM blind spot:

- Models may recommend call/cc generators without understanding dynamic
  cleanup interactions. The source explicitly marks that as unsafe for some
  finally patterns.

*** 11.04.3.10 Syntax helper DSLs for macro ergonomics

Source: =.data/gerbil-utils/syntax.ss=.

Key features:

- Macro-facing helpers separate keyword and positional syntax arguments.
- =stx-apply= and =stx-call= apply functions to syntax argument lists.
- =defsyntax-stx= and =defsyntax-stx/form= wrap syntax application patterns.
- =begin-syntax= parsing helpers analyze formals, keyword defaults, optional
  arguments, and rest arguments.

Harness meaning:

- Macro ergonomics can be factored into reusable syntax helpers instead of
  repeated low-level syntax-case code.
- A guide topic should teach "syntax helper extraction" for repeated macro
  argument parsing.

LLM blind spot:

- Models often inline macro parsing repeatedly. Gerbil projects can build small
  macro DSLs over syntax objects and formals.

*** 11.04.3.11 Peekable iterators and generic protocol wrappers

Source: =.data/gerbil-utils/peekable-iterator.ss=.

Key features:

- A peekable iterator wraps standard iterator state with cached current value
  and status.
- Status distinguishes empty cache, value, exception, and end.
- =:peekable-iter= is generic and has methods for iterator, pairs, null,
  vector, string, hash table, procedure, port, and object.
- Cached exceptions are replayed and finalization is coordinated through
  iterator fini.

Harness meaning:

- Stateful wrappers can still be high-quality when the state machine is named,
  small, and method-dispatched.
- Policy should distinguish "scattered mutation" from "state machine under a
  protocol wrapper".

LLM blind spot:

- Models may miss that exception caching and finalization are semantic states,
  not incidental implementation details.

*** 11.04.3.12 Build DSL and environment lifecycle utility

Source: =.data/gerbil-utils/building.ss=.

Key features:

- Build scripts are normalized through one environment initializer.
- The file coordinates source directory, load path, path-config root,
  application source/home directories, version path, dependency metadata,
  entry points, compiler options, pkg-config, and Nix detection.
- It uses =defrules= for build initialization and =define-entry-point= for
  CLI-facing commands.

Harness meaning:

- This is not primarily a language-feature exemplar; it is an engineering
  utility exemplar.
- The useful policy lesson is lifecycle consolidation: environment setup
  should have one owner and one public entry contract.

LLM blind spot:

- Models may copy individual build snippets. The real value is the centralized
  lifecycle boundary.

** 11.04.4 What Is Still Missing From Harness Evidence

The current harness already has policy labels for typed-combinator style,
macro runtime-source witness, functional data transform, controlled branch
shape, and engineering comment quality. It is not complete enough yet.

Missing or weak evidence categories:

- runtime-source selector resolution to =query --code=
- runtime-source fact registry coverage for match, MOP, control, actors,
  parameters, continuations, and syntax helpers
- phase-aware import/export facts in ordinary search packets
- macro helper extraction facts: repeated =syntax-case= argument parsing,
  keyword syntax splitting, generated identifiers
- control-flow owner classification: =dynamic-wind=, =let/cc=, =call/cc=,
  continuation capture, coroutine, actor, parameterization, locks
- stateful protocol wrapper classification: small state machine under
  =defstruct= plus generic methods
- bootstrap artifact filtering: generated =bootstrap/*.scm= stubs should not
  outrank source =src/gerbil/**/*.ss= for engineering audits
- project import capability facts for advanced libraries such as autocurry or
  generator packages

** 11.04.5 Optimization Plan

*** 11.04.5.1 P0: Close runtime-source selector-to-code

Fix =asp gerbil-scheme query --selector gerbil-runtime-source://... --code= so
selectors emitted by =search runtime-source= can be resolved without falling
back to manual file windows.

Acceptance:

- =search runtime-source macro sugar module-sugar= emits selector lines.
- The emitted selector can be passed unchanged to =query --selector ... --code=.
- Output is only source comments plus the complete source form.

*** 11.04.5.2 P0: Teach ranking to prefer source over bootstrap stubs

When the query asks about engineering or language implementation, generated
=bootstrap/gerbil/*.scm= wrappers should not outrank source
=src/gerbil/**/*.ss= owners unless the user explicitly asks for bootstrap
output.

Acceptance:

- Queries for MOP, match, module, syntax, control, or thread return
  =src/gerbil/.../*.ss= owners before generated stubs.
- Search output labels generated/bootstrap facts separately.

*** 11.04.5.3 P1: Add an advanced-feature fact catalog

Add a catalog that maps parser evidence to feature categories:

- =hygienic-syntax-object=
- =phase-aware-module=
- =mop-class-type-generation=
- =extensible-match-dsl=
- =runtime-control-boundary=
- =actor-continuation-diagnostics=
- =autocurry-application-semantics=
- =typed-combinator-style=
- =generator-control-inversion=
- =syntax-helper-dsl=
- =stateful-protocol-wrapper=
- =build-lifecycle-owner=

Acceptance:

- =asp gerbil-scheme search pipe= can surface these categories in compact
  evidence.
- =guide --code --topic <category>= returns one source-backed exemplar.
- =--more= returns adjacent exemplars without flooding the agent.

*** 11.04.5.4 P1: Refine policy severity

Use warning-level guidance for most style and engineering advice. Reserve hard
gates for evidence integrity and source-backed selector closure.

Warning-level:

- missing algebraic contract comments on reusable helpers
- macro owner lacks hygiene/phase/boundary rationale
- repeated macro argument parsing without helper extraction
- manual destructuring where match facts fit
- continuation or dynamic-wind owner lacks lifecycle comment
- stateful wrapper lacks explicit status/state invariant

Harder gate:

- guide =--code= emits non-source wrappers
- runtime-source search emits a selector that cannot be resolved
- policy claims macro/phase/MOP facts without parser or runtime-source witness

*** 11.04.5.5 P1: Add guide topics from this audit

Guide topics should be backed by real source examples, not inline invented
samples:

- =hygienic-syntax-object=
- =phase-aware-module=
- =mop-class-type-generation=
- =extensible-match-dsl=
- =runtime-control-boundary=
- =actor-continuation-diagnostics=
- =autocurry-semantics=
- =generator-control-inversion=
- =syntax-helper-dsl=
- =stateful-protocol-wrapper=
- =build-lifecycle-owner=

Each default topic should emit one exemplar. =--more= should add a second
exemplar. =--level advanced= can include cross-file relationships such as
module-sugar plus expander/module, or generator plus runtime/control.

*** 11.04.5.6 P2: Add compare/report surfaces

Add comparison views that help agents understand why a sample belongs to one
category rather than another:

- runtime implementation vs utils style exemplar
- pure combinator transform vs control-flow owner
- protocol wrapper vs scattered mutation
- macro helper DSL vs one-off macro transformer
- module phase transform vs ordinary import list

Acceptance:

- =asp gerbil-scheme search compare engineering-axis runtime utils --view seeds=
  or an equivalent provider-declared shape returns compact evidence and next
  guide topics.

** 11.04.6 Policy Translation

The harness should avoid rigid style policing. The right shape is:

- parser collects owner kind, feature markers, import phases, control markers,
  state markers, comments, and source selectors
- policy emits warnings with source-backed examples
- guide =--code= returns only real source comment plus complete source form
- agents decide the local rewrite using the project owner context

This lets the harness guide agents toward high-quality Gerbil without blocking
valid low-level runtime or parser code.

** 11.04.7 Bottom Line

The complete picture is stronger than "use higher-order functions".

=.data/gerbil= shows that Gerbil engineering depends on hygienic syntax
objects, phase-aware modules, MOP-generated type metadata, extensible pattern
DSLs, runtime control boundaries, actor continuations, and source-aware
diagnostics.

=.data/gerbil-utils= shows advanced language usage: autocurrying,
multi-value composition, algebraic comments, generator/control inversion,
syntax-helper DSLs, protocol wrappers, and lifecycle-oriented build utilities.

The harness should now optimize for evidence quality:

- close runtime-source selector-to-code
- rank source files over generated stubs
- add advanced feature facts
- expand guide topics with real source exemplars
- keep style rules warning-oriented and flexible

That gives agents a reliable loop: detect the feature from parser evidence,
show one precise source-backed exemplar, explain the engineering boundary, and
then let the model write idiomatic code for the local owner.

** 11.04.8 Parser Closure Audit

This follow-up audit checked the report against the current native parser and
the real =.data/gerbil-utils= checkout.

Validation command:

#+begin_src sh
GERBIL_LOADPATH=src:t gxi -e '(begin (import :parser/core) ...)'

#+end_src

Observed parser coverage:

- =.data/gerbil-utils= collection succeeds with =files=99=,
  =definitions=956=, =calls=6393=, and =parseErrors=0=.
- High-order facts are produced for =cut=, =curry=, =compose=, =rcompose=,
  =!>=, =!!>=, =case-lambda=, =stx-apply=, =generating<-for-each=,
  =generating<-list=, =generating<-vector=, =generating-fold=,
  =generating-partition=, =generating<-cothread=, =in-cothread=,
  =in-cothread/peekable=, =:peekable-iter=, =peekable-iterator-peek=,
  =peekable-iterator-next!=, sequence folds/maps/predicates, and
  =with-list-builder=.
- Control-flow facts are produced for =let/cc=,
  =##continuation-capture=, =make-parameter=, =parameterize=, =spawn=,
  =spawn/name=, =yield=, =cothread=, =in-coroutine=, =try=, =catch=,
  =finally=, =with-catch=, =match=, and builder boundaries.
- Macro facts are produced for =define-syntax=, =defsyntax=,
  =defsyntax-for-match=, =defrule=, and =defrules=.
- POO facts are produced for =class=, =generic=, and =method=.

Closed parser gaps from this audit:

- Value-style function definitions now expose callable shape when the value is
  =lambda= or =case-lambda=. This covers gerbil-utils examples such as
  =(def curry (case-lambda ...))= and =(def rcurry (case-lambda ...))=.
- Value-style partial applications now expose callable arity when the value is
  =cut= or =cute=. This keeps function-valued helpers such as
  =(def bump (cut + <> 1))= aligned with adjacent algebraic contracts.
- Parser snapshots now include definition =arity= next to =formals=. This makes
  callable-shape evidence visible to snapshot/report consumers instead of
  leaving it only inside policy checks.
- Export DSL items now have parser-owned =moduleExport= facts with
  =modifier=, =alias=, =module=, =symbols=, and selector evidence. These facts
  are projected through parser snapshots, JSON, and the structural search index.
- POO method overloads are no longer treated as duplicate type bindings.
  Multiple =defmethod= entries under one generic are legal overload evidence,
  not duplicate owner definitions.
- Bare Gerbil metadata forms such as =prelude: :<root>= and
  =package: gerbil/core= now parse as source-file metadata instead of only
  being tolerated as atom forms.
- Runtime module-sugar macro definition heads =defsyntax-for-import=,
  =defsyntax-for-export=, and =defsyntax-for-import-export= now produce
  parser-owned definition, macro, formal, and phase evidence. This closes the
  =only-in= family shape used by the active Gerbil runtime source.
- Runtime-source guide code now resolves acquisition =sourceExamples= through
  =gerbil-runtime-source://<source-path>#<symbol>= selectors into active
  runtime source snippets. =guide --code --topic macro-runtime-source= emits
  source only, prioritizes examples with =sourceComments= evidence, and uses
  =--more= to add the next runtime-source exemplar without repeating the first.

Remaining frontier:

- =gerbil-utils= check still reports warning-level policy findings: =R008= has
  28 POO extension-shape warnings and =R013= has 20 typed-combinator-style
  warnings. These are not parse failures; they are agent-facing guidance for
  external generic/method extension patterns and test-file contract coverage.
