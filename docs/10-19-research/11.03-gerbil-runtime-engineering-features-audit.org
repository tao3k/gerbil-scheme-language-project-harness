** 11.03.1 Scope

This note audits the version-matched =.data/gerbil= source and selected
=.data/gerbil-utils= files for advanced Gerbil/Scheme engineering features.
The goal is to give =gerbil-scheme-language-project-harness= a source-backed
quality baseline for policy guidance, code examples, and future guide output.

This is not a request to copy upstream implementation into the harness. The
useful artifact is the engineering shape:

- hygienic macro code is syntax-object and phase aware
- module imports and exports are expansion-time contracts
- DSLs such as match, import/export sugar, iteration, and builders are normal
  engineering surfaces
- high-order utilities compress semantic transforms into small combinators
- comments should carry algebraic contract shape and boundary rationale, not
  prose that repeats the code

** 11.03.3 Advanced Features Found

*** 11.03.3.1 Hygienic syntax objects, not text substitution

=src/gerbil/runtime/syntax.ss= shows that Gerbil macro work is built on syntax
objects and AST values. The relevant engineering signals are:

- syntax carries source, context, marks, and phase-like data
- syntax errors preserve contextual irritants, source, context, phi, and marks
- matchers operate through =syntax-case=, =with-syntax=, identifiers, lists,
  AST pairs, and generated identifiers
- identifier comparison uses hygienic equality operations such as bound or free
  identifier equality instead of string equality

This means a harness policy must not treat macro code as textual rewriting.
When an agent edits macro owners, it needs to reason about identifier scope,
source locations, phase, marks, and generated names.

Policy implication:

- Warn when macro-bearing owners use comments that only describe surface
  syntax and omit hygiene, generated identifier, or phase boundaries.
- Prefer parser-owned macro facts over lexical keyword checks.
- Require guide examples for macro code to include the source comment and the
  complete function or macro form, without extra rendered wrappers.

*** 11.03.3.2 First-class macro expanders

=src/gerbil/core/macro-object.ss= defines macro objects that carry an expander
and apply it through the core expander path. The file imports runtime support
at a positive phase, which is a concrete reminder that macro engineering lives
across expansion phases.

This differs from plain function abstraction. A macro owner is a compile-time
participant with an expansion contract.

Policy implication:

- A macro object, =defsyntax=, or =defrules= owner should produce guidance
  about its expansion-time contract.
- Macro code should be checked with an expansion witness when practical, not
  only runtime execution.

*** 11.03.3.3 Core forms are implemented as syntax transformers

=src/gerbil/core/sugar.ss= implements familiar forms through syntax
transformers. It normalizes =lambda= variants, definition forms, conditionals,
and compilation-target conditionals through syntax matching and syntax-object
utilities.

The important lesson is that Gerbil style does not reserve macros for rare
metaprogramming tricks. Core ergonomic language forms are built by disciplined
syntax transformers.

Policy implication:

- Harness guidance should treat macros as normal engineering assets when they
  compress a repeated semantic shape.
- The warning target is not "macro exists"; the target is macro code without a
  clear expansion contract, hygiene story, or test witness.

*** 11.03.3.4 Phase-aware module and import/export DSLs

=src/gerbil/core/module-sugar.ss= implements import and export forms such as
=for-syntax=, =for-template=, =only-in=, =except-in=, =rename-in=,
=prefix-in=, and export variants through expansion-time transformers.

This is one of the biggest differences from simple Scheme code. Import/export
forms are not just file-level declarations. They are semantic DSLs that rewrite
module surfaces with phase, rename, filter, and grouping meaning.

Policy implication:

- Module-owner comments should identify the import/export transformation shape
  when phase or rename behavior is nontrivial.
- A parser fact for phase-bearing import forms should be stronger than a raw
  text search for =import= or =export=.
- Advice should name the semantic operation: filter, rename, prefix, group,
  template phase, or syntax phase.

*** 11.03.3.5 Extensible pattern matching as a macro DSL

=src/gerbil/core/match.ss= shows that match is not only a convenience form. It
is an extensible pattern DSL backed by macro objects, syntax-local values,
identifier equality, class/type information, algebraic pattern forms, and
applicative destructuring.

High-signal engineering features include:

- user-extensible match macros
- syntax-local lookup of match behavior
- equality-sensitive identifiers
- class and structure pattern support
- algebraic pattern composition such as and, or, and not
- applicative destructuring patterns

Policy implication:

- When harness code hand-rolls nested conditional destructuring, the guide can
  suggest a match-oriented rewrite if parser evidence shows a stable pattern
  shape.
- This should be warning-level advice, not a hard failure, because some
  low-level parser paths may need explicit control flow.

*** 11.03.3.6 Generic iteration, comprehensions, and cleanup boundaries

=src/std/iter.ss= exposes iteration as a generic protocol with methods for
common sequence and stream-like inputs. It also provides for-style forms,
collecting folds, yielding, and cleanup behavior around iterators.

The engineering difference from ad hoc recursion is the named protocol:

- sequence traversal is abstracted behind a standard iterator surface
- collection and fold forms express intent directly
- cleanup and end-of-stream behavior are part of the protocol
- coroutine-like iteration exists as a first-class control-flow option

Policy implication:

- Manual loops over common iterable structures should be classified as a design
  opportunity when a standard iterator or comprehension form would express the
  same shape more clearly.
- The harness should surface this as "consider iterator/comprehension protocol"
  with source-backed examples, not as a rigid ban on loops.

*** 11.03.3.7 Builder macros hide mutation behind a small contract

=src/std/misc/list-builder.ss= provides builder-style construction where local
mutation is hidden behind a narrow push/peek contract. The design is useful
because it keeps the performance boundary explicit while preserving a compact
external API.

This is the correct model for performance-sensitive Scheme code:

- local mutation is acceptable when the boundary is named and small
- the builder owns ordering and tail updates
- the user-facing body stays focused on semantic production
- comments document the continuation or mutation boundary rather than repeat
  each assignment

Policy implication:

- Mutation inside a named builder/control abstraction should be treated
  differently from scattered mutation in ordinary code.
- A policy warning should ask for boundary comments when local mutation is
  hidden for performance.

*** 11.03.3.8 Gerbil-utils high-order combinator style

=.data/gerbil-utils/list.ss= and related utility code show a compact,
function-oriented style that is closer to Scheme/Haskell engineering than to
imperative loop-heavy code.

The strongest recurring pattern is the three-part style:

- algebraic contract comments describe the transform shape
- small functions expose one semantic combinator at a time
- optimization comments explain why a special branch exists

Examples include curried function specializers, fold/reduce/map adapters,
list-monad style helpers, deduplicating builders, and C3 linearization helpers.
The comments are valuable because they tell an agent the type-like shape of the
function before it reads the implementation.

Policy implication:

- The harness should encourage algebraic contract comments for exported or
  reusable functions.
- It should not force one exact comment grammar. The important requirement is
  that the comment exposes the function's input/output transformation shape.
- One-line compressed comments are weak when they mix contract and boundary
  rationale. Prefer a short contract line plus a separate boundary line when
  both are needed.

** 11.03.4 How Gerbil Differs From Basic Scheme For Engineering

Gerbil has ordinary Scheme forms, but its engineering style is stronger than
"let plus cond plus recursive loops".

- Macros are hygienic syntax-object programs, not token replacement.
- Phases are explicit in real code through imports and compile-time expanders.
- Module imports and exports are transformed by syntax-level DSLs.
- Pattern matching is extensible and can consume local macro/type/class facts.
- Object, class, generic, and protocol surfaces can participate in macro and
  runtime design.
- Iteration and builders give common control-flow and performance needs a
  named abstraction.
- Utilities often encode transforms as small composable functions with
  lightweight type-like comments.

The harness should therefore guide agents toward Gerbil-native engineering:
macro contracts, phase comments, composable transforms, and narrow performance
abstractions.

** 11.03.5 Policy And Guide Direction

*** 11.03.5.1 Parser-owned facts to collect

Future policy work should prefer native parser facts for these signals:

- owner kind: =def=, =defrules=, =defsyntax=, =defclass=, =defmethod=,
  =defstruct=, =defgeneric=, and related forms
- macro body markers: =syntax-case=, =with-syntax=, generated identifiers,
  syntax-local lookup, and identifier equality
- phase imports: positive and negative =phi= imports plus for-syntax and
  for-template sugar
- module import/export transforms: only, except, rename, prefix, group, and
  export variants
- high-order intent: =compose=, =cut=, =curry=, map/fold/reduce adapters, and
  function-returning functions
- builder/control abstractions: =with-list-builder=, yield, iterator, coroutine
  style, and cleanup-bearing loops
- comment blocks near owners: contract shape, boundary rationale, optimization
  rationale

The parser should provide evidence, not final aesthetic judgment. The policy
layer can then choose warning/advice severity.

*** 11.03.5.2 Warning-level rules

These should be warning-level because agents can use the guidance without the
harness rejecting valid low-level code:

- macro owner lacks expansion/hygiene/phase guidance
- module owner with phase or rename transforms lacks import/export rationale
- reusable function lacks an algebraic contract-style comment
- manual loop or nested conditional destructuring appears where parser facts
  suggest a stable match, fold, or iterator shape
- local mutation appears outside a named builder or lifecycle abstraction
- contract and boundary comments are compressed onto one line and are hard to
  scan

*** 11.03.5.3 Harder gates

Harder gates should be rare and tied to objective evidence:

- generated guide example cannot be traced back to a real source selector
- macro policy reports facts from generic memory instead of version-matched
  runtime source
- guide =--code= output adds renderer wrappers around code instead of emitting
  the source comment plus the complete source form
- parser claims a phase, import/export transform, or macro owner without a
  native AST/source witness

** 11.03.6 Guide Output Contract

Agent-facing guide examples should be progressive:

- default guide: one source-backed exemplar for the requested topic
- =--more=: additional exemplars when the agent asks for broader comparison
- =--code=: only the original source comment and complete source form

Good topics to expose from this audit:

- =hygienic-macro=
- =phase-aware-module=
- =match-dsl=
- =iterator-protocol=
- =builder-boundary=
- =typed-combinator-style=

Each topic should include a short reason that tells the agent what to copy:
contract shape, phase boundary, hygiene boundary, optimization boundary, or
composition style.

** 11.03.7 Bottom Line

The source baseline is clear. High-quality Gerbil code is not just shorter
Scheme. It uses syntax-object hygiene, expansion phases, import/export DSLs,
extensible pattern DSLs, generic iteration, builder abstractions, and compact
high-order combinators.

The harness should use those facts to steer agents through warning-level
guidance and source-backed examples. The right policy shape is flexible but
evidence-rich: collect native parser facts, show one precise exemplar, explain
the engineering boundary, and leave enough room for the agent to write idiomatic
code for the local owner.
