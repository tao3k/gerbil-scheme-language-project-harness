#+TITLE: Gerbil build.ss Engineering Guidelines
#+DATE: 2026-06-25

* Purpose

This note fixes the harness rule for package-level =build.ss= files:
=build.ss= is a thin package build entrypoint, not a second build system.

The package build owner should delegate source discovery, dependency
compilation, and worker scheduling to Gerbil's native build APIs such as
=std/make=, =std/build-script=, and =clan/building=. Package-specific code may
describe build specs, but it should not hand-write broad file walks, duplicate
test execution, or shell/native compiler orchestration.

* Build Target Rules

- Keep =build.ss= small. Use it to configure package roots and call reusable
  build helpers.
- Prefer one coherent =make= graph per module naming root. Do not split one
  logical test target into independent =make= calls unless the phases have
  different module roots or output contracts.
- When a package stores modules under =src= or =t=, do not fake a package-root
  graph if imports are written for a root-specific module namespace.
- Let =gxtest-main= execute tests. It should not own build specs, source
  discovery, or policy collection.
- Do not precompile entire test-support trees by directory name. Top-level
  test modules should define the test scope, and imports should remain part of
  the native compile graph.
- Keep full-project policy gates explicit. Ordinary test targets should run
  policy through the gxtest scope they are already executing.

* Test Target Shape

The canonical harness test target is:

1. Resolve package root once.
2. Build each required module root at most once.
3. Build top-level =t/*-test.ss= entry files, not whole support trees.
4. Run a harness-owned gxtest runner over top-level =t/*-test.ss= entry
   files.

This keeps compile scheduling inside Gerbil's native make graph while avoiding
a second pass over =t/unit=, =t/snapshot=, and =t/policy= as separate support
trees.

The harness runner must preserve upstream =gxtest= semantics for each file.
It may parallelize top-level files by running isolated =gxi= subprocesses, but
it must keep output grouped by file and must not mutate =std/test= global state
from multiple threads in one process.

Default runner parallelism is derived from the machine: =build-worker-count=,
capped by the number of top-level test files. =build-worker-count= is the same
resource boundary used by package builds: =GERBIL_BUILD_CORES= when configured,
otherwise the host CPU count.

* Anti-Patterns

- =build.ss= writes shell launchers or embeds shell pipeline control.
- =build.ss= manually invokes =gxc=, =gsc=, =gxi=, =cc=, or =clang= for package
  targets instead of using Gerbil build APIs.
- =build.ss= defines local worker queues, build-job runners, phase receipts, or
  stamp freshness on top of an already-selected native =std/make= /
  =clan/building= surface.
- =build.ss test= compiles source modules, build support modules, test support
  modules, and gxtest modules through separate broad =make= calls.
- A normal test target runs the same policy surface twice: once through gxtest
  and once through a separate full-project gate.
- A runner parallelizes by invoking =run-test-suite!= concurrently inside one
  process. Use file-level subprocess isolation instead.

* Performance Gate

Changes to =build.ss= or =build-support/gslph-build.ss= should preserve two
separate receipts:

- dirty/cold target time, which includes compilation;
- warm target time, which should expose test execution and policy/schema
  overhead without hiding behind compile cost.

The receipt must identify the phase that regressed. A single aggregate
=gxi build.ss test= duration is not enough to guard this boundary.

The harness emits machine-readable lines:
* Next Improvement Plan: Native Make Alignment
This plan extends the current =build.ss= rules with the upstream evidence from
=.data/gerbil/src/std/make.ss= and =.data/gerbil/src/tools/gxpkg.ss=.  The goal
is not another wrapper around Gerbil builds; the goal is to make the harness
thin enough that native Gerbil build semantics remain the owner of scheduling,
incremental freshness, release flags, and test execution scope.

** Package-General Acceptance Gate
Optimization work is only accepted when it improves the reusable package
contract. A change must pass these gates before it becomes policy or a build
API:

- The owner must be an upstream Gerbil owner (=std/make=, =std/build-script=,
  =gxpkg=, or =clan/building=) or a reusable harness API consumed by arbitrary
  packages.
- The implementation must not key behavior on a downstream project name, fixed
  binary name, fixed worker count, fixed source directory beyond a declared
  root, or a local shell wrapper.
- The scenario must use a neutral fixture package, not =gslph=, =poo-flow=, or
  another real downstream package as the only proof.
- The repair message must give one concrete package-general rewrite or command,
  so an agent can map its own package intent onto the native build surface.
- Performance receipts must name the native phase that improved. A faster
  aggregate command is not evidence if it hides duplicate test execution or a
  bypassed build owner.

This gate is the guard against optimizing for one current repository shape. If
a fix cannot be explained through these package-general terms, it belongs in a
local experiment, not in the harness policy.

** Upstream Evidence To Preserve
- =std/make#make= parameterizes =current-directory=, adds the source directory to
  the load path, calls =make-build=, and only flushes pending compile jobs when
  =settings-parallelize= is greater than one.
- =std/make#make-build= normalizes the build spec once, then builds a native
  dependency graph with import and dependency mutexes, completion tables,
  timestamp tables, a build barrier, a work channel, and =build-worker-count=.
- =std/make#build?= and =build-dependent?= own freshness. They compare declared
  inputs, declared outputs, imported module dependencies, and library
  timestamps. A harness layer must not replace this with broad directory walks
  or local stamp files.
- =std/make= starts one coordinator per spec, pushes ready specs to the work
  channel, closes the channel after the barrier clears, and joins worker
  threads. Harness parallelism should compose with this boundary, not duplicate
  it.
- =gxpkg#pkg-build= passes =--release=, =--optimized=, and =--debug= to
  =build.ss compile= through Gerbil build environment variables. The harness
  build CLI should expose policy and receipt surfaces without re-owning these
  native package build options.

** Workstream A: Thin Downstream Build API
- Keep downstream =build.ss= as a declaration surface: package name, module
  roots, top-level test files, and optional harness policy attachment.
- Replace any downstream broad source collection API with an API that reads the
  same build/test roots the native build already uses. =asp check --full= may
  still run full-project policy, but ordinary =gxi build.ss test= must stay
  scoped to the files and import closure that =gxtest= is executing.
- Add a scenario where a package has =src= modules, =t= top-level tests, helper
  modules, and one changed dependency. The expected result is a single native
  build graph and policy coverage for the gxtest import closure, not a second
  full-project pass.

** Workstream B: Native Incremental Freshness
- Introduce a fixture that mutates one imported module timestamp and verifies
  that the harness reports the native =std/make= rebuild boundary instead of
  invalidating the whole source tree.
- Add a negative fixture that writes local cache/stamp freshness logic in
  =build.ss=. This should trigger the existing package-build policy family,
  because freshness is owned by =std/make=.
- Extend benchmark receipts with explicit warm-path phases for native build
  freshness, gxtest execution, policy collection, and policy evaluation. The
  sub-second path must be visible in =observedTimings= instead of hidden behind
  one aggregate =gxi build.ss test= duration.

** Workstream C: Test Scope Without Duplicate Execution
- Make the gxtest integration the single normal test path: build top-level
  =t/*-test.ss= files once, run them once, and attach harness policy to the same
  file scope and import closure.
- Add a scenario that fails if the same top-level test file is executed once by
  upstream gxtest and again by a separate policy test target.
- Keep full-project policy as an explicit command outside ordinary package test
  targets. Its receipt should say it is full-project, so it cannot be confused
  with normal gxtest policy coverage.

** Workstream D: Release/Optimized Build Surface
- Treat =gxpkg build -R -O= and the =GERBIL_BUILD_RELEASE= /
  =GERBIL_BUILD_OPTIMIZED= environment path as the native owner for static and
  optimized executable builds.
- The harness build CLI may validate, document, and forward these options, but
  it must not manually invoke =gxc=, =gsc=, =cc=, or local shell scripts for
  package build products.
- Add a scenario proving that =build.ss compile= receives the native release and
  optimized option path while policy receipts remain harness-owned.

** Scenario And Gate Additions
- Extend =package-build-framework-overreach= with cases for local freshness
  stamps, local worker queues, and local release/optimized compiler invocation.
- Extend =downstream-gxtest-policy-scope= with a multi-root fixture whose warning
  lives behind a test import, and with a receipt that proves the test file ran
  once.
- Add a native-incremental fixture with before/after =observedTimings= measured
  in milliseconds and with a warm-path regression budget.
- Add a build-cli fixture that validates native =gxpkg= option forwarding without
  creating a second binary or shell wrapper contract.

** Success Criteria
- Downstream =build.ss= files remain thin and declarative; source discovery,
  dependency scheduling, freshness, and release/optimized compilation stay in
  native Gerbil build APIs.
- =gxi build.ss test= produces one normal test execution path and one policy
  attachment path over the same gxtest scope.
- Warm fixture receipts expose sub-second timings for build freshness, gxtest,
  policy collection, and policy evaluation.
- The harness policy can explain the repair with one concrete command or one
  concrete rewrite example, without hardcoding a downstream project name.
