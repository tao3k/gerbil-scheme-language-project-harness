** 11.01.1 Source

Research source: the version-pinned upstream Gerbil source from
~https://git.cons.io/mighty-gerbils/gerbil~ at commit ~add92248~.

Gerbil is a Scheme dialect on top of Gambit. The user-facing tools in
~src/gerbil/main.ss~ include ~gxi~, ~gxc~, ~gxpkg~, ~gxtest~, ~gxtags~,
~gxprof~, and ensemble/httpd helpers. The harness is implemented in Gerbil and
uses Gerbil reader/expander APIs as the parser boundary.

** 11.01.2 Native Facts

- Source files are primarily ~.ss~, ~.ssi~, and ~.scm~.
- Package roots commonly contain ~gerbil.pkg~ and ~build.ss~.
- ~gerbil.pkg~ ~depend:~ entries are treated as required ~gxpkg~
  dependencies by the harness packet. The current Poo extension support is
  activated only from that declared dependency fact; optional dependency
  semantics need explicit package-manager evidence before they are surfaced.
- Poo extension support currently means packet-level capability facts for
  declared ~gerbil-poo~ dependencies. It does not yet parse Poo-specific class,
  protocol, or metaobject forms as separate semantic facts.
- Module headers use plain top-level fields such as ~prelude:~, ~package:~,
  and ~namespace:~.
- Dependency edges are visible in top-level ~(import ...)~ forms with Gerbil
  module ids like ~:std/error~ and relative string imports like
  ~"core/runtime"~.
- Public surface is visible in ~(export ...)~, but macro expansion is required
  for complete phase-aware export facts.
- Common owner items are introduced through ~def~, ~defstruct~, ~defclass~,
  ~defsyntax~, ~defrules~, ~defalias~, ~defmethod~, ~defcompile-method~,
  ~define~, ~define-values~, and ~define-syntax~.

** 11.01.3 First-Pass Harness Boundary

1. Build a workspace index from ~.ss~, ~.ssi~, ~.scm~, ~.sld~, and
   ~gerbil.pkg~.
2. Use ~core-read-module~ for ordinary Gerbil modules so module id, prelude,
   namespace, source locations, and body forms come from the native expander.
3. Use ~read-syntax-from-file~ or syntax-only ~#lang~ fallback when importing a
   project prelude would execute project reader code during search.
4. Extract package/prelude/namespace/import/export/include/definition facts.
5. Expose the common search/query/check/agent surfaces used by the other
   language harnesses.
6. Keep compiler/type enrichment as the next step: use ~import-module~,
   module-context exports, and compiler facts only where the command explicitly
   needs expanded or phase-aware semantics.

** 11.01.4 Compiler Proof Boundary

The upstream compiler evidence in ~.data/gerbil/src/gerbil/compiler~ supports
our current lightweight proof design, not a full proof system.

- ~compiler/optimize-base.ss~ defines optimizer metadata such as ~!signature~,
  ~!class~, ~!predicate~, and ~!type-subtype?~.  The subtype relation is
  metadata-level: exact type equality, top type, procedure shape, and
  class/subclass checks through the class precedence list.
- ~compiler/optimize-ann.ss~ uses local assertion environments for match and
  syntax-case optimization.  Predicate assertions are folded into local
  ~env-type~ facts, and ~assert-type~ can satisfy positive assertions only when
  class/subclass metadata proves the requested relation.
- There is no general constraint solver, quantifier reasoning, principal type
  inference, proof-term calculus, or cross-module theorem prover in this layer.

Harness implication: =TypeProof= should remain a positive derivation witness
over normalized =TypeSpec= relations.  It can expose rule chains for policy
composition, but it must not claim complete type-theory soundness.  If we need
stronger guarantees later, the next step is a separate typed core plus
counterexample diagnostics, not stretching Gerbil compiler optimizer facts into
a theorem prover.

** 11.01.5 Medium-Weight Compiler Evidence Landing

The harness now exposes this upstream compiler boundary through an explicit
=compiler-evidence= search namespace:

#+begin_src sh
gerbil-scheme-harness search compiler-evidence optimizer subtype assertion --workspace . --view seeds
gerbil-scheme-harness search proof subtype record alias --workspace . --view seeds
#+end_src

The =compiler-evidence= packet is intentionally ordinary language evidence,
not a =runtime-source-acquisition= packet.  It points agents at the versioned
runtime source selectors that justify a local derivation:

- ~gerbil-runtime-source://src/gerbil/compiler/optimize-base.ss#!signature~
- ~gerbil-runtime-source://src/gerbil/compiler/optimize-base.ss#!class~
- ~gerbil-runtime-source://src/gerbil/compiler/optimize-base.ss#!predicate~
- ~gerbil-runtime-source://src/gerbil/compiler/optimize-base.ss#!type-subtype?~
- ~gerbil-runtime-source://src/gerbil/compiler/optimize-base.ss#!class-subclass?~
- ~gerbil-runtime-source://src/gerbil/compiler/optimize-ann.ss#fold-assert-type~
- ~gerbil-runtime-source://src/gerbil/compiler/optimize-ann.ss#assert-type~

Agent-facing rule: type/proof repairs may cite =compiler-evidence= only as
medium-weight evidence.  Valid claims are normalized contract parsing,
metadata-backed subtype checks, local assertion environment facts, and source
selector witnesses.  Invalid claims remain full type theory, quantifier
reasoning, proof-term calculus, principal inference, and global theorem proving
over module imports.

The =proof= namespace is the stable medium-weight proof system surface.  It
serializes TypeSpec proof trees from =src/types/subtyping.ss= with root rule,
recursive premises, depth, node count, preorder rule chain, and compiler
evidence linkage.  It is mature enough for agent repair gates because the
packet is schema-backed and generated from the actual proof engine, but it
remains intentionally below a complete theorem prover.
