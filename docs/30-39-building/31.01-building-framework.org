* 31.01 Building Framework

The harness now treats building as a peer framework to testing. Testing owns
test objects, runners, and policy feedback. Building owns stage objects, build
plans, std/make adapters, receipts, and performance gates.

This is not a replacement for Gerbil's =std/make=. The correct boundary is:
=std/make= remains the execution engine, while the harness Building Framework
adds stage semantics, package-specific ordering, receipt-oriented observability,
and stable performance gates around that engine.

  Tests["gxtest + performance gates"]

#+end_src

** 31.01.1 Reflection: upstream improvement

The upstream improvement is architectural, not just a local helper.

Before this change, harness package compilation was effectively a flat
=std/make= call over a package build spec. That made two things hard:

- The package API had no named stage boundary, so dependency-order mistakes
  surfaced as compile races instead of as plan-level errors.
- Build behavior lived under =build-api=, so there was no reusable library
  equivalent to the existing testing library.

The new upstream shape is:

- =src/testing= remains the testing framework.
- =src/building= becomes the building framework.
- =src/build-api= becomes the harness/package adapter that consumes
  =src/building=.

This gives downstream projects a stable library surface instead of forcing them
to copy package build scripts. It also lets future policy checks reason about
building as a first-class harness capability.

** 31.01.2 Reflection: building-system improvement

The building improvement is the split between execution and orchestration.

=std/make= is good at compiling Gerbil specs. The missing layer for agent-facing
engineering is not another compiler; it is an object model around build stages.
That model now has:

- =build-stage= for label, kind, spec, current predicate, runner, after hook,
  and description.
- =build-stage-receipt= for label, kind, status, description, result, and
  elapsed jiffies.
- =build-stage-receipt->alist= and =build-plan-receipts->alist= for
  machine-readable agent projection.
- =build-plan-run!= for ordered stage execution.
- =std-builder= for adapting a stage spec to =std/make=.
- =default-std-builder= for the ordinary Gerbil =std/make= runner.

The harness package build now uses =gslph-package-api-stage-specs= instead of a
single flat list for the package API. The new =building/model.ss=,
=building/std-builder.ss=, and =building/facade.ss= files are explicit ordered
stages. This fixes the dependency race where a facade could compile before the
module it imports. The stage spec plan is cached inside the build process so
agent-facing repeated plan inspection does not repeatedly scan and sort source
directories.

#+begin_src mermaid
flowchart TB
  S1["building/model.ss"]
  S2["building/std-builder.ss"]
  S3["building/facade.ss"]
  S4["types/parser/policy stages"]
  S5["commands/build-api stages"]

  S1 --> S2 --> S3 --> S4 --> S5

The important design rule is:

- =std/make= compiles one stage.
- =src/building= decides what a stage means.
- =src/build-api= decides which harness package stages to run.

** 31.01.3 Reflection: speed and developer UX

The speed improvement is not only raw milliseconds. It is also faster diagnosis
and safer iteration.

Developer UX improvements:

- Agents can inspect a named stage instead of reverse-engineering a flat list.
- A compile failure can be attached to a stage label such as
  =building/std-builder.ss= or =package-api=.
- Stage receipts create a natural observability path for future JSON output.
- The build path no longer requires =clean= before every build; it is compatible
  with incremental Gerbil package builds.
- The Homebrew path can stay normal: no =GERBIL_LOADPATH= escape hatch and no
  Nix SDK unsetting wrapper.
- =SDKROOT=, =CC=, and related toolchain variables are inherited unchanged.
  A caller owns explicit overrides; the harness never identifies a package
  manager, compiler installation, or SDK path by string prefix.
- Performance gates can measure framework overhead separately from full compiler
  cost.

The current micro gate intentionally measures framework overhead:

- 2000 skipped stage runs must stay under 200 ms.
- 500 stale stage runs through the std-builder path must stay under 300 ms.
- 500 package stage plan constructions must stay under 300 ms.

Current local measurement on this machine:

#+begin_example
skipStageMs         1.469
runStageMs          0.500
packageStagePlanMs  0.651
#+end_example

Before stage-spec caching, 500 package stage plan constructions measured about
240 ms locally. After caching, the same gate measured about 0.651 ms. These
numbers are not a final benchmark suite. They are the first regression floor:
if framework dispatch or plan inspection becomes expensive, the test fails
before the cost is hidden inside a full package compile.

** 31.01.4 Current evidence

The current landed evidence is:

- =src/building/model.ss= defines stage and receipt objects.
- =src/building/model.ss= projects stage receipts into alists for agent-facing
  observability.
- =src/building/std-builder.ss= adapts stages to =std/make= and exports
  =default-std-builder=.
- =src/building/facade.ss= exports the public building library.
- =src/build-api/framework.ss= keeps the old build-api framework path as a
  facade over =src/building=.
- =src/build-api/package-spec.ss= includes explicit ordered building stages.
- =src/build-api/package-spec.ss= caches the package stage plan inside the
  build process.
- =build.ss= sends every =compile= and =spec= variant through
  =src/build-api/native-build.ss=; it has no direct =package-build.ss= path.
- =src/build-api/native-build.ss= runs package and native targets through
  =default-std-builder=, =build-profile=, =build-request=, and
  =build-request-run!=.
- =src/testing/gxtest-build.ss= delegates selected source and test compilation
  to the native Building adapter instead of dynamically loading a package
  compiler.
- =build.ss= bootstraps the package environment before dynamically loading the
  GxTest runner; this keeps the runner's native Building dependency outside the
  script expansion path.
- Selected GxTest receipts project the two Building requests as =buildPlan=
  metadata, so an agent can distinguish source and test stages after a run.
- =src/build-api/package-build.ss= owns only package-local Gerbil environment
  setup and the build lock; =src/building/std-builder.ss= is the sole
  =std/make= execution owner.
- =src/build-api/native-build.ss= records the build plan projection in package
  build receipt metadata when the default Building path runs.
- =t/building-framework-test.ss= covers the new public building API.
- =t/build-api-framework-test.ss= protects the build-api facade.
- =t/building-performance-test.ss= establishes the first executable performance
  gate.
- =t/building-gxtest-stage-boundary-test.ss= protects the root test command's
  selected-receipt Building projection.
- =t/scenarios/building/std-builder-stage-boundary/benchmark.ss= records the
  benchmark contract and current observed timings.

** 31.01.5 API shape

The intended downstream shape is compact:

#+begin_src scheme
(import :gslph/src/building/facade)

(def builder
  (default-std-builder
   "src"
   [optimize: #f
    parallelize: 4
    prefix: ".gerbil/lib"]))

(def stages
  [(std-builder-stage
    builder
    "core"
    ["core.ss"]
    (lambda (stage context) #f))
   (std-builder-stage
    builder
    "facade"
    ["facade.ss"]
    (lambda (stage context) #f))])

(build-plan-run! stages 'package)
#+end_src

This is intentionally similar in spirit to testing: users should import one
facade, construct ordinary Scheme values, and run a framework function.
