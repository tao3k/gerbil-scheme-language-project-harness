** Core Claim

Building is not a replacement compiler and must not claim to make a stale
=std/make= compilation faster.  Gerbil remains authoritative for compilation
and =gxpkg= remains authoritative for package management.  The improvement is
above that layer: typed policy values make a build request inspectable before
execution, testable through a shared framework, and observable after execution.

** Shared Model

#+begin_src mermaid
flowchart LR
  Scenario["Existing building scenario benchmark"] --> Contract["Benchmark contract"]
  Request["BuildRequest"] --> Adapter["src/building/testing.ss"]
  Adapter --> Case["Testing performance-case"]
  Contract --> Case
  Case --> Suite["Testing performance-suite"]
  Suite --> Receipt["Testing receipt"]
  Request --> Plan["pure BuildStage plan"]
  Plan --> Warm["zero std/make calls when planning"]
#+end_src

The adapter reuses =t/scenarios/building/std-builder-stage-boundary/benchmark.ss=.
There is one scenario fixture and one benchmark contract; Building does not
create a parallel performance format.  It projects a =BuildRequest= into the
Testing Framework's POO =performance-case=, =performance-suite=, and
=testing-project= objects.

** What Improves Upstream

1. *Incremental work elimination.* A current request is represented by a
   predicate and projects =skipped= receipts without calling =std/make=.  The
   speed gain is avoiding unnecessary compiler work, not optimizing the
   compiler's stale path by assertion.
2. *Reusable policy.* =BuildProfile= moves builder options, labels, and hooks
   out of a package script and into a value that many requests can share.
3. *Standardized feedback.* Testing receives a validated benchmark fixture and
   returns a structured receipt.  Agents can distinguish fixture failure,
   runner failure, invalid result, and elapsed time instead of parsing shell
   text.
4. *One quality gate.* Scenario contract validation, runner result validation,
   warm-path no-make assertion, and existing building performance budgets are
   exercised together.

** Improvement Matrix

| Layer | Upstream authority | Framework addition | Developer and agent evidence |
|-------+--------------------+--------------------+------------------------------|
| =std/make= | Compile stale sources | None: it remains the compiler | No false compiler-speed claim |
| =gxpkg= | Resolve and build packages | None: it remains package authority | Normal package build receipt remains valid |
| Build API | Currentness and durable receipts | Package-stage planning boundary | Agent can tell current, stale, and skipped states |
| Building | No shared request policy value | Reusable =BuildProfile= and pure =BuildRequest= plan | One warm request projects ordered =BuildStage= values with zero =std/make= calls |
| Testing | Generic scenario runner and receipt | Adapter from =BuildRequest= to =performance-case= | One existing scenario validates fixture, result, timing, and receipt together |

This is a standards and feedback improvement for downstream projects: they keep one scenario contract, receive one structured receipt vocabulary, and can gate the actual warm control path without manufacturing a second benchmark format. The performance claim is therefore measurable work avoidance and earlier diagnostic feedback, never a claim to replace or out-run upstream compilation.

** Boundary and Non-Claims

- =std/make= executes real compilation; =gxpkg= resolves and builds packages.
- Build API owns package currentness and durable package receipts.
- Building owns profile/request values and pure stage-plan construction.
- Testing owns benchmark fixture validation, measurement, performance receipts,
  and suite-level status.

The framework therefore adds control-plane standardization and warm-path
efficiency.  It does not duplicate Gerbil's compilation algorithm or promise
that a real stale compilation is inherently faster than upstream.

** Downstream Surface

No raw alist DSL is introduced at the user boundary: callers compose ordinary
BuildRequest and Testing Framework POO objects.  The scenario fixture remains
the declarative contract and can be selected by the existing test runner.

** Bootstrap and Native Toolchain Boundary
#+begin_src mermaid
flowchart LR
  Bootstrap["serial bootstrap"] --> Model["building/model"]
  Model --> Builder["building/std-builder"]
  Builder --> Facade["building/facade"]
  Facade --> Declarative["building/declarative"]
  Declarative --> Requests["parallel BuildRequest stages"]
  Requests --> Receipt["active gxpkg environment receipt"]
#+end_src
The bootstrap is intentionally =parallelize: 1=: each module publishes an SSI consumed by the next module. This is an interface-order constraint, not a machine-capacity decision. Once the Framework is available, package requests use =sync-build-worker-count!= so independent stages use the available worker budget.

=NativeToolchain= is the explicit boundary for the C compiler environment. Its default clears inherited =SDKROOT= and =DEVELOPER_DIR= values that can cross-contaminate a native Gerbil installation; callers can construct a toolchain object with their own values. The receipt and output root are both resolved from the active =gxpkg= environment, so a linked dependency never writes its state into the Harness source checkout.

** Performance Gates

1. The existing building scenario fixture supplies benchmark contract fields,
   resource limits, measurement phases, and quality axes.
2. The canonical Testing Framework scenario test selects =building-stage-plan=
   by case name, consumes the fixture through the shared runner, and yields one
   valid performance receipt.
3. The runner only invokes =build-request-stage-plan=; its validator requires
   every output to be a =BuildStage=, and the test asserts zero =std/make=
   calls.
4. Building's existing warm, stale, stage-plan, package-plan, and request-plan
   budgets remain separate micro gates.  The integration suite prevents those
   micro gates from drifting away from the user-facing scenario contract.
