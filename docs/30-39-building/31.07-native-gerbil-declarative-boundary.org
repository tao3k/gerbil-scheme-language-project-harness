** Decision

The harness must not create a build or test CLI. It improves the Gerbil
experience only above the upstream execution boundary:

- =gxi= evaluates the source-root =src/build.ss= directly. The file is not an
  executable shell program and needs no executable permission.
- =clan/building= owns the native multicall entrypoint, =meta=, =spec=, and
  =compile= actions. It delegates the build graph to =std/make=.
- =gxtest= owns test-file discovery, =-test= export discovery, suite selection,
  setup/cleanup, and process exit status.
- =gxpkg env= may supply the local package environment to =gxi= and =gxtest=.
  It is not a second build command and does not dispatch =build.ss=.

The Building and Testing Framework remains a library. It may define data,
receipts, scenario fixtures, and assertion helpers. It must not reimplement
upstream argument parsing, source discovery, test discovery, build scheduling,
or process dispatch.

** Source-Root Declaration

Runtime modules live under =src/=. The only build declaration therefore lives
at =src/build.ss= and uses the public upstream macro:

#+begin_src scheme
;;; -*- Gerbil -*-
(import :clan/building)

(init-build-environment!
  spec: (lambda () '(... standard std/make build specification ...)))
#+end_src

=init-build-environment!= sets the native source root from its own file,
installs the upstream multicall entrypoint, and accepts a declarative spec
procedure. There is no package-root wrapper, executable-bit requirement, or
private upstream API use.

The project command is:

#+begin_src sh
gxpkg env gxi src/build.ss compile
#+end_src

An FFI package may keep a native =std/make= build spec as data. It may not
turn that into manual compiler or process dispatch.

** Package Module Identity

The source package declares =gslph/src=.  Every package-local runtime import
must therefore use =:gslph/src/<module>=.  A source-relative import such as
=:support/time= is not a valid package identity: it only works when a caller
adds =src/= to =GERBIL_LOADPATH=, and native =std/make= correctly rejects it.

The migration is parser-first.  Native module import facts identify imports;
the source-owner map decides whether a module is local.  Only an import whose
unprefixed module corresponds to =src/<module>.ss= is rewritten to
=:gslph/src/<module>=.  Upstream namespaces such as =:std/=, =:gerbil/=,
=:clan/=, and =:wr/= remain unchanged.  Existing =:gslph/src/= imports and
dynamic module names are already canonical.

** Test Contract

Tests remain ordinary modules with gxtest-compatible =-test= exports. The only
user-facing test forms are native ones:

#+begin_src sh
gxpkg env gxtest t/...
gxpkg env gxtest -r '^policy$' t/...
gxpkg env gxtest t/package-build-test.ss
#+end_src

The Testing Framework may supply assertions, scenario fixtures, and
performance receipt checks. It does not parse a second test CLI or make a
selected file become a broad project run. Suite selection uses gxtest's =-r=
option; file and recursive scope use its normal arguments.

** Framework Boundary

=src/building/= retains typed =BuildProfile= / =BuildRequest= data only where
it improves inspection, receipt construction, or scenario tests. It lowers to
or observes upstream =std/make= work; it is never the package executor.

=src/testing/= retains gxtest-compatible helpers and scenario assertions. Its
performance gate is a gxtest case. There is no =bench= command and no
framework-specific test-runner compatibility layer.

=Justfile= may name repeatable developer workflows, but every recipe invokes
an upstream command directly. A build recipe uses =gxpkg env gxi=; a test
recipe uses =gxpkg env gxtest=. It does not rewrite source paths or command
behavior.

On Darwin, a repository environment can inject a Nix SDK while the installed
Gerbil runtime invokes the host C compiler. The installer may therefore make
one command-local host-toolchain selection: clear inherited =SDKROOT= and
resolve =CC= with =xcrun --find clang= for the native =gxi= invocation. This
is not a compiler dispatcher, SDK path, or persistent environment mutation:
the values are not stored, no path is hard-coded, and =gxi= still owns
compilation. Other platforms invoke the same upstream command without an
adaptation.

** Required Deletions

The migration removes:

- the former root =build.ss= dispatcher and its executable-bit requirement;
- bootstrap stage lists, source-interface materialization, and dynamic loading;
- handwritten getopt, =compile= / =install= dispatch, and test subcommands;
- SDK paths, compiler paths, load-path rewriting, and package-manager-specific
  shell dispatch; a command-local Darwin host-toolchain selection is permitted
  only to prevent an externally injected SDK from being mixed with the
  installed Gerbil runtime;
- package-local wrappers which only forward to gxtest or std/make.

=native-build.ss= and =gxtest-runner.ss= are retained only where they expose a
real reusable library concern. Any remaining part that serves the removed
command protocol is deleted rather than preserved as compatibility code.

** Migration And Verification

1. Put the upstream =clan/building= declaration in =src/build.ss=.
2. Change build recipes to =gxpkg env gxi src/build.ss compile= and test recipes
   to direct =gxpkg env gxtest= invocations.
3. Replace root test selections with direct gxtest file, recursive, and
   suite-regex arguments while preserving selected-file scope.
4. Add a downstream scenario that invokes a source-root build through =gxi= and
   test modules through =gxtest= using only public library helpers.
5. Extend the build policy to reject handwritten command routing, dynamic
   loading, persistent or hard-coded environment rewriting, and direct compiler
   process dispatch, while accepting the upstream =clan/building= form,
   standard =std/make= FFI data, and the narrowly-scoped Darwin host-toolchain
   selection.
6. Add a parser-fact package-identity gate: no source-relative import may map
   to an owned source module after migration.
7. Keep performance checks as gxtest cases, with declaration/receipt overhead
   measured separately from native compiler work and with explicit units.

** Acceptance Criteria

- =gxpkg env gxi src/build.ss compile= builds without a harness command or
  executable permission.
- =gxpkg env gxtest t/...= runs the full suite, and an explicit file runs only
  that file.
- =src/build.ss= stays a small upstream declaration with no custom runner or
  mutable bootstrap logic.
- Downstream examples use the same native commands and no hidden wrapper.
- Policy and scenario coverage reject reintroduction of a second command
  protocol.
