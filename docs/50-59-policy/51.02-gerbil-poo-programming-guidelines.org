* 51.02 Gerbil POO Programming Guidelines

This document is an agent-facing guide for writing better Gerbil POO code. It
is not a hard restriction list. Policy findings should steer agents toward the
source-backed POO shape that keeps behavior compositional, inspectable, and easy
to repair.

** 51.02.1 Source Evidence

The reference source is =.data/gerbil-poo=.

- =doc/prototypes.md:1-13= defines the programming model as incremental
  composition of code and data fragments with late binding.
- =doc/prototypes.md:162-182= separates an object as a queryable instance from
  a composable prototype with open computations. The instance is the fixed
  point of those computations.
- =doc/poo.md:299-319= treats =.mix= as the way to instantiate a prototype when
  side effects or repeated instantiation are needed.
- =t/object-test.ss= and =t/mop-test.ss= show the concrete syntax agents should
  learn: ={(:: @ super) slot: ...}=, =>=, =>.+=, =?=, and =.mix=.

- =object.ss:149-151= implements =object<-alist= by converting each pair into a constant slot spec before calling =make-object=. This is an external alist adapter, not the preferred declaration form for stable config/profile objects.
- =object.ss:128-131= and =object.ss:414-416= show that =.get= expands into recursive =.ref= calls, and =.ref= resolves through the object instance hash. Repeated projection groups should be bound once or batched.
- =object.ss:390-401= implements =.o= as both expression syntax and match syntax; the match side expands through =o?/slots= and =.refs/slots=.
- =object.ss:204-218= exposes =(.refs/slots slots)= as the lower-level batch projection helper behind slot predicates and match-style projections.
- =object.ss:356-368= implements =with-slots= and =def-slots= by binding named slots through =.@=. Fixed local slot sets should use this native binding form instead of materializing a dynamic projection list.
The harness should teach those shapes before it warns about drift.

** 51.02.1.1 Optimizer-Visible POO Shape

POO optimization should use Gerbil and Gambit language features instead of
rewriting native POO into ad hoc data structures. The repair target is code the
compiler can see clearly:

- fixed slot sets become lexical bindings with =with-slots=,
- compact config specs use =lambda-match= or local destructuring helpers,
- stable object declarations stay as named native =.o= values,
- full-object materialization happens once at a boundary,
- hot loops carry scalar/list/hash state rather than repeated POO construction,
  projection, or traversal.

This is the same principle as the non-POO optimizer scenarios: make the checked
public boundary explicit, then keep the internal hot path direct, lexical, and
small. POO policy should therefore prefer optimizer-visible native POO over both
basic generated Scheme scaffolding and adapter-heavy rewrites.

** 51.02.2 Positive Shape

Good POO code starts from a prototype and refines slot computations inside that
prototype:

#+begin_src scheme
(def base-order
  {id: 0
   total: 0
   meta: {currency: 'USD}})

(def order
  {(:: @ base-order)
   id: ? 1
   total: => 1+
   meta: =>.+ {source: 'agent}
   status: ? 'draft})

(def order-view
  (.mix order))
#+end_src

The important property is not the surface syntax alone. The fixed point remains
inside one POO shape:

- =?= declares defaults without forcing an outer constructor to read and copy
  every slot.
- =>= transforms an inherited slot computation.
- =>.+= merges nested object/prototype fragments.
- =.mix= materializes an instance when the prototype should be instantiated.

Large configuration, profile, and schema objects keep the same rule. The
performance shape is not to rewrite native POO into =object<-alist= or a hash.
Name the stable native =.o= shape once, compose a small native overlay at the
boundary for genuinely dynamic slots, and keep hot loops in scalar state after
capturing needed slots outside the loop:
#+begin_src scheme
(def +profile-shape+
  (.o kind: "marlin.config-interface.loop-policy.profile-projection-module.v1"
      profile-id: "default"
      loop-limit: 100
      timeout-ms: 5000))

(def (build-profile module-id-value)
  (.mix +profile-shape+
        (.o module-id: module-id-value)))

(def (profile-budget profile rows)
  (let ((loop-limit (.ref profile 'loop-limit))
        (timeout-ms (.ref profile 'timeout-ms)))
    (let loop ((rest rows) (count 0) (budget 0))
      (if (or (null? rest) (>= count loop-limit))
        budget
        (loop (cdr rest) (+ count 1) (+ budget timeout-ms))))))
#+end_src
This keeps the object shape native and inspectable while avoiding repeated slot
resolution, repeated composition, or repeated object construction inside the
loop.
When the slot set is static in one local computation, bind the slots with
=with-slots= and reduce scalar state once. Use =.refs/slots= only when the slot
names are dynamic or the projection itself is the boundary protocol:
#+begin_src scheme
(def (profile-slot-total profile)
  (with-slots (loop-limit timeout-ms retry-budget priority-weight)
      profile
    (+ loop-limit timeout-ms retry-budget priority-weight)))

(def (profile-budget profile rows)
  (let (slot-total (profile-slot-total profile))
    (let loop ((rest rows) (budget 0))
      (if (null? rest)
        budget
        (loop (cdr rest) (+ budget slot-total))))))
#+end_src
For large config-interface objects that need a boundary metadata projection, prefer a native =.o= match pattern over repeated =.get= calls:
#+begin_src scheme
(def (loop-pattern config)
  (let (loops-policy (UserInterfaceMarlinLoopsPolicy config))
    (match loops-policy
      ((.o loop-name: loop-name-value
           summary: summary-value
           workspace-root: workspace-root-value
           budget: budget-value
           isolation: isolation-value
           maker: maker-value
           checker: checker-value
           level: level-value
           priority: priority-value
           source: source-value
           control-plane-owner: control-plane-owner-value
           owner: owner-value)
       (make-loop-pattern-descriptor
        loop-name-value
        summary-value
        `((level . ,level-value)
          (priority . ,priority-value)
          (watched-scope . (,workspace-root-value))
          (budget . ,budget-value)
          (isolation . ,isolation-value)
          (maker . ,maker-value)
          (checker . ,checker-value)
          (metadata .
                    ((acting_on . ,workspace-root-value)
                     (source . ,source-value)
                     (module-system . ,control-plane-owner-value)
                     (policy-owner . ,owner-value)))))))))
#+end_src
This is the native POO projection idiom for a stable object crossing into an alist-style boundary payload: keep the source object as =.o= and let the =.o= match pattern express the projection. Use =.refs/slots= directly only when a lower-level batch projection function is needed.
** 51.02.3 Allowed Boundary Reads

=.ref=, =.@=, and =.get= are valid POO APIs. The harness must not teach agents
to avoid them globally.

Use boundary reads for:

- small accessors,
- assertions,
- adapters that leave the POO object boundary,
- test or diagnostic projections.

Example:

#+begin_src scheme
(def (order-id source)
  (.ref source 'id))
#+end_src

That shape is intentionally allowed. The quality issue appears when a
constructor repeatedly projects slots from a POO object and rebuilds a parallel
hash/alist/object outside the prototype.

** 51.02.4 R026 Repair Contract

=GERBIL-SCHEME-AGENT-POLICY-026= is a soft warning. It should fire only for a
constructor/build function that performs a projection burst with =.ref=, =.@=,
or =.get=.

Drift shape:

#+begin_src scheme
(def (make-order source)
  (let ((id (.ref source 'id))
        (total (.ref source 'total))
        (currency (.ref source 'currency)))
    (hash (id id) (total total) (currency currency))))
#+end_src

Repair shape:

#+begin_src scheme
(def order
  {(:: @ base-order)
   id: ? 1
   total: => 1+
   meta: =>.+ {source: 'agent}})
#+end_src

The finding details must keep this contract visible:

- =guidanceMode= is =soft-warning=.
- =allowedUse= explains that isolated boundary reads are valid.
- =repairShape= points to prototype-local composition.
- =docsPath= points back to this document.

** 51.02.5 Agent Repair Loop

When an agent receives R026:

1. Keep legitimate boundary accessors.
2. Find the repeated slot projection group in the constructor/build function.
3. Introduce or reuse a base prototype.
4. Move defaults, transforms, and nested merges into one ={(:: @ super) ...}=
   body.
5. Use =.mix= only when an instance must be materialized.
6. Re-run the scenario or policy check and require R026 findings to clear.

The scenario owner is
=t/scenarios/policy/poo-prototype-fixed-point/=. Its =input= side demonstrates
the projection burst. Its =expected= side demonstrates the repair shape while
preserving a valid =order-id= boundary read.

** 51.02.7 Scenario Corpus

The fixed-point constructor path remains the only R026 warning. The broader POO
layers are now covered as a soft guidance corpus under
=t/scenarios/policy/poo-*= and =t/snapshots/policy-poo-guidance-corpus.ss=.
Those scenarios deliberately record parser-owned facts and target findings
without adding new hard policy rules.

The corpus covers these repair and false-positive boundaries:

1. =poo-type-descriptor= starts from raw slot validation code and repairs to a
   =Class.= descriptor with =Slot= method slots. Parser facts must expose
   =descriptor:mop= and =descriptor:class-slot=.
2. =poo-method-family-serialization= starts from scattered json/string/bytes
   helpers and repairs to =methods.string<-json= / =methods.bytes<-marshal=
   slots. Parser facts must expose =methodFamily:serialization=.
3. =poo-algebra-wrapper= starts from local map/wrap/bind helpers and repairs to
   =Functor.= and =Wrapper.= descriptors. Existing R013/typeclass guidance owns
   style warnings; this corpus only asserts POO facts and no POO target
   findings.
4. =poo-domain-algebra= starts from one-off polynomial helpers and repairs to a
   parameterized =Polynomial.= descriptor over a ring slot. Parser facts must
   expose =domainAlgebra:polynomial-ring= and
   =domainDescriptor:parameterized-ring=.
5. =poo-boundary-accessors= keeps =.ref=, =.@=, and =.get= boundary reads legal,
   including =build-*= and =new-*= helpers with a single projection. This guards
   against turning R026 into a hard ban on POO access APIs.

Future rules must first pass this corpus as soft guidance evidence. A new
warning is acceptable only when parser-owned facts identify the shape precisely
enough to avoid rejecting valid POO use. The goal is better POO, not narrower
POO.
