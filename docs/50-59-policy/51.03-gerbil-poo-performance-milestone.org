* Status

This milestone records the first broad Gerbil POO performance hardening pass for
the language-project harness. The outcome is a cost model, policy coverage, test
scenarios, and remaining research lanes for downstream agents that write POO
code.

The milestone is not a Gerbil upstream implementation change. The harness
enforces agent-facing guidance so downstream projects learn the efficient POO
shape before repeated hot-loop patterns enter source code.

* Evidence Boundary

The research used ASP search-first discovery against the gerbil-poo checkout and
then microbenchmarks in the same checkout. Source evidence was taken from
parser-owned selectors and search metadata, not raw source scans.

Primary owner evidence:

- =object.ss= owns object construction, instance cache, materialization, clone,
  composition, projection, and mutation boundaries.
- =mop.ss= owns MOP type validation, lens update behavior, and
  =MonomorphicObject=/=Function= type construction.
- =fq.ss= owns finite-field type construction, including parameterized =F_q=
  descriptors.
- =number.ss= owns numeric type construction, including fresh =Z/= and
  =IntegerRange= descriptors plus cached =UIntN=/=IntN= descriptors.
- =doc/poo.md= and =t/object-test.ss= remain supporting documentation and test
  owners, but the policy cost model is implementation-owned.

Key selector evidence:

- =object.ss:454-474= shows =.put!= as a direct mutation boundary and =.cc= as
  carbon-copy clone override setup.
- =object.ss:188-215= shows =.for-each!=, =.alist=, =.alist/sort=,
  =force-object=, and =hash<-object= full-object iteration and
  materialization paths.
- =object.ss:209-229= shows =o?/slots= and =.refs/slots= projection helpers.
- =mop.ss:420-484= shows =Lens .modify= delegates to =.set= and =slot-lens=
  implements =.set= with =.cc=.
- =mop.ss:188-240= shows =MonomorphicObject= and =Function= construct
  fresh POO type objects.
- =fq.ss:24-95= shows =F_q= constructs a fresh finite-field type object.
- =number.ss:90-160= shows =IntegerRange= and =Z/= construct fresh numeric
  type objects.
- =number.ss:175-207= shows =UIntN= and =IntN= use =hash-ensure-ref= and are
  cached constructors.

* Cost Model

The policy model separates POO cost into ten hot-loop surfaces.

** Optimizer Visibility

The POO performance rules are now tied to Gerbil optimizer-visible repair
shapes. A passing repair should not merely remove one expensive call. It should
make the hot path lexical and stable enough for the compiler model used by the
broader language scenarios:

- =with-slots= and =.o= match patterns expose fixed slot access instead of
  dynamic projection in every iteration.
- =lambda-match= and small local destructuring helpers keep large declaration
  data compact without hiding calls behind generic tables.
- one boundary materialization or construction step leaves the loop with
  scalar/list/hash state.
- native =.o= remains the source shape for stable profiles and configs.

The benchmark contract field =optimizerVisibility= records which
optimizer-friendly shape each POO scenario is asserting. The field is part of
the scenario contract for high-yield POO scenarios, not documentation-only
commentary.

** Construction

POO object construction is appropriate at module, boundary, and meaningful
shape-change points. It is expensive when repeated in hot loops over stable
input data.

Covered patterns:

- large native =.o= shape declaration with a small dynamic overlay
- loop-local =object<-alist=
- loop-local =object<-hash=
- loop-local =object<-fun=
- loop-local =make-object=

Scenario coverage must keep those adapter and low-level constructor variants
visible. The repair target is still native POO: name the stable =.o= shape,
carry scalar/list/hash loop state while iterating, and construct or compose a
POO object once at the boundary. Do not replace native POO declarations with
=object<-fun= or hand-written =make-object= as a style rewrite.

Harness rules:

- =GERBIL-SCHEME-AGENT-POLICY-027=: native =.o= profile/config declarations
  are preserved; optimize by naming stable native shapes, composing only small
  boundary overlays, and keeping loop state scalar after boundary slot capture.
- =GERBIL-SCHEME-AGENT-POLICY-033=: stable object construction should be hoisted out
  of loops or represented as scalar/list/hash loop state and built once.

** Clone And Override

=.cc= is pure and idiomatic for one-time refinement, but it walks override data
and produces a new object shape. In loops, it should usually become scalar loop
state plus one final =.cc=.

Covered patterns:

- loop-local =.cc=
- loop-local lens modification through =slot-lens=

Harness rules:

- =GERBIL-SCHEME-AGENT-POLICY-028=: repeated =.cc= clone overrides should move out of
  hot loops.
- =GERBIL-SCHEME-AGENT-POLICY-032=: repeated =Lens .modify= over =slot-lens= should
  move to scalar accumulation plus one final object update.

** Materialization And Projection

Full-object materialization and full-object iteration are correct at
serialization and inspection boundaries. Slot projection is also valid, but
repeated stable projection or traversal in a loop wastes work.

Covered patterns:

- loop-local =.for-each!=
- loop-local =.alist=
- loop-local =.alist/sort=
- loop-local =.all-slots=
- loop-local =.all-slots/sort=
- loop-local =.refs/slots=
- loop-local =hash<-object=
- loop-local =force-object=

Scenario coverage should include all full-object traversal and materialization
variants, not just one list adapter. The optimized shape is a named native
object plus one boundary snapshot for each required representation; the loop
then consumes scalar/list/hash state without forcing the POO object again.

Harness rule:

- =GERBIL-SCHEME-AGENT-POLICY-029=: materialize, iterate, or project once outside
  the loop, or carry only the required scalar state.

** Composition

POO composition with supers is the right boundary for building object behavior.
Repeated composition in a loop creates new supers chains and pushes lookup cost
to later slot access.

Covered patterns:

- loop-local =.mix=
- loop-local =.extend=
- loop-local =.+=
- loop-local =object<-alist=, =object<-hash=, or =object<-fun= with =supers:=

Harness rule:

- =GERBIL-SCHEME-AGENT-POLICY-030=: compose once at the boundary after scalar
  accumulation.

** Validation

Validation is a boundary operation. Repeated validation over the same stable
object shape inside a loop walks object values repeatedly.

Covered patterns:

- loop-local =validate=
- loop-local =element?=
- loop-local =monomorphic-object?=

Harness rule:

- =GERBIL-SCHEME-AGENT-POLICY-031=: validate once before the loop or validate only
  changed scalar fields.

** Type Construction

MOP type construction is also a boundary operation. =MonomorphicObject= builds a
fresh type object with closure slots, =Function= validates input/output type
lists before building a descriptor object, =F_q= builds a finite-field
descriptor with a nested =Z/pZ= descriptor, and =Z/= / =IntegerRange= build
fresh numeric descriptors, so stable type objects should be named and reused.

Covered pattern:

- loop-local =MonomorphicObject=
- loop-local =Function=
- loop-local =F_q=
- loop-local =Z/=
- loop-local =IntegerRange=

Harness rule:

- =GERBIL-SCHEME-AGENT-POLICY-034=: hoist stable POO/MOP type construction to a named
  binding outside the loop.

** Debug Instrumentation

Debug instrumentation is a setup boundary. =trace-poo= walks all slots and
installs traced wrappers, so repeated setup in a loop is expensive and also risks
turning ordinary reads into diagnostic output.

Covered pattern:

- loop-local =trace-poo=

Harness rule:

- =GERBIL-SCHEME-AGENT-POLICY-035=: hoist =trace-poo= outside the loop and reuse the
  traced object.

** Slot-Spec Mutation

Slot-spec mutation changes object shape. =.def!= and =.putslot!= are valid setup
tools, but they are not the same as updating a slot value. In hot loops, value
updates should use =.put!= for intentional mutation or scalar state plus one
final pure object update.

Covered patterns:

- loop-local =.def!=
- loop-local =.putslot!=
- loop-local =.setslot!=

Harness rule:

- =GERBIL-SCHEME-AGENT-POLICY-036=: keep slot definitions at setup boundaries; use
  =.put!= for mutable values or a final =.cc= for pure updates.

** Slot Predicates

Slot existence predicates are cheaper than value materialization, but multi-slot
predicate checks still walk a slot list and call =.slot?= repeatedly. When object
shape is stable, the predicate result should be computed once at a boundary.

Covered pattern:

- loop-local =o?/slots=

Harness rule:

- =GERBIL-SCHEME-AGENT-POLICY-037=: hoist stable =o?/slots= predicate results outside
  the loop; hoist only the predicate closure when the object can change but the
  slot list is stable.

* Implementation Boundary Walk

This pass also walked the major POO implementation boundaries, including
boundaries that did not produce a new policy rule. The goal is to keep the
policy model broad enough to explain why a candidate is covered, rejected, or
deferred.

|----------+--------+---------+---------------|
| Object lifecycle | =object.ss= | Construction, instance cache, clone, materialization, projection, composition, and mutation all converge here. | Covered by R027, R028, R029, R030, R033. |
| MOP/type/lens | =mop.ss= | Validation, =Lens .modify=, =slot-lens=, and =MonomorphicObject= type construction are separate runtime cost surfaces. | Covered by R031, R032, R034. |

used ASP =rg=, =fd=, and selector queries for source evidence.

The benchmark receipts below are intentionally small, repeatable probes. They
measure warm behavior inside one Gerbil process and exclude command startup from
the compared loop body.

  =1910ms= inside the loop versus =0ms= for a hoisted =.alist= sum reused by
  the loop.
- =Function= type construction: 100000 loop constructions measured =1017ms=
  versus =8ms= for a hoisted type descriptor reused by the loop.
- =F_q= finite-field type construction: 100000 loop constructions measured
  =524ms= versus =5ms= for a hoisted type descriptor reused by the loop.
- =Z/= modular integer type construction: 100000 loop constructions measured
  =392ms= versus =12ms= for a hoisted type descriptor reused by the loop.
- =IntegerRange= type construction: 100000 loop constructions measured
  =739ms= versus =4ms= for a hoisted type descriptor reused by the loop.

* Negative Findings

The research also rejected several candidate policies.

** Cached Slot Reads

Stable repeated =.ref= is not currently a structural policy target. The object
instance cache and slot function cache mean repeated slot reads are far cheaper
than clone, construction, validation, or materialization surfaces. Hoisting can
still help in extremely tight loops, but a broad rule would over-report normal
POO code.

** Assertion Checks

=check-assertions!= is not active in the inspected instantiation path. The
implementation has a commented assertion hook, so assertion checks are not a
current hot-loop surface for harness policy.

** JSON Conversion

JSON conversion searches did not identify a distinct implementation-owned
hot-loop surface beyond existing materialization and validation paths. The
candidate was rejected to avoid policy duplication.

** Method Dispatch And Tables

Method dispatch/table searches converged on =doc/poo.md=, =table.ss=, and
=t/table-testing.ss= but did not produce an implementation hot range with
loop-local repeated dynamic dispatch cost. The boundary is documented as walked,
but it should not become a rule without a concrete downstream call-site and a
microbenchmark.

** Table Serialization
=table.ss= owns generic table and set projection through =.list<-=,
=.json<-=, =.bytes<-=, and =.marshal=. These are collection serialization
boundaries over a typeclass, not POO object slot traversal. Do not fold them
into R029 unless a future parser fact can distinguish stable table projection
inside a hot loop from intentional collection boundary conversion.

** Dynamic =.call=

=.call= searches converged on documentation and did not produce a runtime
hot-loop implementation surface with =loop=, =repeated=, and =hoist= evidence.
The API remains a normal POO method invocation surface. If downstream evidence
shows a stable method closure repeatedly looked up inside a loop, revisit it as
a narrow call-site rule rather than a broad =.call= warning.

** Cached Slot Reads

=.ref/cached= is an intentional low-level API for reading an existing cached slot
value. It should not be reported as a performance issue by itself. The caution is
semantic: it returns the default when a value has not been computed and can
observe stale cache if object shape was mutated after instantiation.

** IO And Marshal

=io.ss= owns generic =marshal= and =unmarshal= boundaries. These are
serialization boundaries, not inner-loop POO object lifecycle operations.
Repeated serialization should normally be fixed by hoisting materialization,
projection, or validation; that keeps this lane under R029 and R031.

** Functional Wrapper Layer

=fun.ss= defines wrapper/typeclass abstractions such as =Wrap.= and =Functor=.
The inspected evidence did not show loop-local construction or repeated wrapper
allocation patterns. This remains a future lane only if downstream evidence
shows stable wrapper type construction inside hot loops.

** Auxiliary Data Structures

=trie.ss=, =fq.ss=, =rationaldict.ss=, =polynomial.ss=, and =table.ss= are
supporting data-structure owners. They can have meaningful internal performance
work, but they are not the same surface as downstream POO object lifecycle
guidance. The trie implementation explicitly documents wrapper-level cached
digests, lazy loading, and cached counts as possible extension points. Those are
upstream/library design lanes, not harness rules for agent-authored POO code.

** Generic Type Constructors

Some parameterized type constructors are explicitly cached, for example
constructors that use =hash-ensure-ref=. R034 is therefore intentionally narrow:
it covers =MonomorphicObject=, =Function=, =F_q=, =Z/=, and =IntegerRange=,
where the inspected implementations construct fresh type objects. Cached
constructors such as =UIntN= and =IntN= remain outside the rule.

* Next Research Lanes

The next broad pass should look beyond per-API policy and into parser/runtime
support:

- derive loop-invariant object/type/projection suggestions from syntax facts
  instead of only matching callee names
- expose performance guidance in search packets so agents see the preferred
  repair command before editing
- add a policy report section that groups all POO performance findings by cost
  surface
- compare gerbil-poo patterns against downstream projects with real loop-heavy
  workloads
- investigate whether gerbil-poo itself should cache selected type constructors
  such as =MonomorphicObject=, =Function=, =F_q=, =Z/=, or =IntegerRange=,
  while keeping the harness rule as downstream guidance
- investigate upstream data-structure optimizations separately from harness
  policy, especially trie wrapper-level cached digests, lazy loading, and cached
  counts
