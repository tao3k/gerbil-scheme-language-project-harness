#+title: 51.01 Gerbil Policy Rules Roadmap

* 51.01 Gerbil Policy Rules Roadmap

** 51.01.1 Rust Harness Reference

The Rust harness policy surface separates modularity rules, agent-facing source
shape rules, and project/test layout rules. The Gerbil harness should follow
that ownership model, but each rule must be backed by native Gerbil parser facts
before it is enabled.

The first implemented Gerbil policy pack mirrors the Rust rule intent without
copying Rust-specific syntax:

- =GERBIL-SCHEME-MOD-R001=: owner facade files named =facade.ss= must not
  contain implementation definitions.
- =GERBIL-SCHEME-MOD-R002=: oversized source leaves with many definitions must
  be split by responsibility.
- =GERBIL-SCHEME-MOD-R003=: sibling source files and owner directories such as
  =src/foo.ss= and =src/foo/= must not share the same owner name.
- =GERBIL-SCHEME-MOD-R004=: files inside an owner directory must not repeat the
  owner name, such as =src/foo/foo.ss=. Use a purpose-named module such as
  =facade.ss= instead.
- =GERBIL-SCHEME-MOD-R005=: executable entrypoints under =bin/= must stay thin
  and must not contain implementation definitions.
- =GERBIL-SCHEME-AGENT-R001=: facade source files must carry an
  agent-readable intent comment near the file header.
- =GERBIL-SCHEME-AGENT-R002=: source owner path segments must avoid generic
  buckets such as =utils=, =common=, =helpers=, and =shared=.
- =GERBIL-SCHEME-AGENT-R003=: unrelated facades must not export the same public
  name.

** 51.01.2 Native Inputs

Policy rules consume =project-index=, =source-file=, =definition=, and file
metadata derived by the native parser. They do not scan command output and do
not own JSON rendering.

The parser now records =source-file-line-count= so the first source-leaf rule
can run from parser-owned file facts. =GERBIL-SCHEME-MOD-R002= follows the Rust
harness threshold shape: it reports files with at least 650 source lines and at
least 40 native definitions.

Facade detection is structural: a file such as =src/foo/facade.ss= is a facade
when the project also contains owner files below =src/foo/=. This matches the
current Gerbil harness layout where =src/parser/facade.ss=,
=src/types/facade.ss=, =src/checker/facade.ss=, and =src/policy/facade.ss= act
as stable facades.

Executable entrypoints follow the same thin-adapter policy. A file such as
=bin/run.ss= may import and dispatch to owned modules, but implementation
definitions belong under =src/= so agent-facing edits land on reusable owners.

** 51.01.3 Check Integration

=run-type-checks/signatures= now includes policy findings after reader, type
environment, and checker findings. The default =check= command therefore uses
the same report packet while allowing policy rules to participate in the full
gate.

The current repository is expected to pass these first policy rules. Positive
coverage lives in =test/policy-test.ss= and builds temporary fixture projects
under =.run= so failing policy fixtures do not pollute normal project scans.

The second policy slice adds generic owner name detection and duplicate facade
export detection. Both consume existing parser facts only: source paths,
facade ownership, and =source-file-exports=.

Self-apply coverage lives in =test/self-apply-test.ss=. It compares the current
harness findings to =test/snapshots/self-apply-findings.ss=. The findings
snapshot is an explicit invariant, not a bare empty list: =status= must be
=pass=, =findingCount= must be =0=, and =findings= must be empty. This means
the harness must pass its own reader, type, checker, and policy rules. The
check report snapshot uses the shared =snapshot/facade= projection so policy
tests and command packet tests exercise the same unit interface.
