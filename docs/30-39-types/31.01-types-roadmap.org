* 31.01 Gerbil Types Roadmap

** 31.01.1 Reference Scope

The =scheme-harness-checker= reference implementation treats type analysis as
a first-class module family:

- JSON schema projection into checker types.
- Type environment construction from signatures and first-pass constraints.
- Type contradiction detection before higher-level rule reporting.
- Termination and whitelist rules that depend on typed facts.

** 31.01.2 Local Boundary

The Gerbil harness uses =src/types/facade.ss= as the stable facade.
Type-specific implementation lives under =src/types/=:

#+begin_src text
src/types/facade.ss
src/types/core.ss
src/types/env.ss
src/types/findings.ss
src/types/model.ss
src/types/signatures.ss
#+end_src

=src/commands/check.ss= remains a command adapter. It asks the types module for
findings and formats the result, but it does not own type facts or type rules.
JSON remains a protocol projection after findings are produced.

** 31.01.3 Environment Facts

=src/types/env.ss= builds a type environment from parser-owned definitions. The
first environment fact records each definition as an =unknown= binding with its
name, kind, parser-owned formals, arity, path, and selector. This keeps the
type pipeline ready for signature and schema-derived types without making the
parser depend on type rules.

=src/types/model.ss= owns native type values. =src/types/signatures.ss= loads
Scheme signature data and merges it into =src/types/env.ss=. Signature data is
native Scheme input; JSON remains only a later projection of these facts.

=src/types/model.ss= now represents structured signature types for pair, list,
vector, union, record, and variadic function forms. The parser accepts native
Scheme signature expressions such as =(list number)=, =(vector string)=,
=(union string number)=, =(record ((value number)) (value))=, and
=(function* number number 1)=. These facts stay in the native type model so
future checker rules can consume typed values instead of reparsing strings.

=src/types/env.ss= also derives parameter-level type facts from function
signatures and parser-owned formals. Fixed-arity function signatures produce
per-parameter bindings when their arity matches the definition; variadic
function signatures can produce repeated parameter bindings after their minimum
arity is satisfied. =any= and =unknown= parameters are not emitted as strong
facts. This keeps the next type-mismatch and termination rules grounded in the
native type environment instead of checker-local string parsing.

The first type rule reports duplicate environment bindings within the same
owner/path as =GERBIL-SCHEME-TYPE-E001=. Names from different source owners stay
distinct.

** 31.01.4 Dispatch Order

The next checker work should flow through the types modules before command or
JSON projection changes:

#+begin_src text
parser facts -> types facts -> types rules -> finding JSON -> check command
#+end_src

Native parser facts are the required test boundary. JSON output is useful for
machines, but it is not the authority for type analysis.
