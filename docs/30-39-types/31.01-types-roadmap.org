* 31.01 Gerbil Types Roadmap

** 31.01.1 Reference Scope

The local reference checkout at =.data/scheme-harness-checker= treats type
analysis as a first-class module family:

- JSON schema projection into checker types.
- Type environment construction from signatures and first-pass constraints.
- Type contradiction detection before higher-level rule reporting.
- Termination and whitelist rules that depend on typed facts.

** 31.01.2 Local Boundary

The Gerbil harness uses =src/types.ss= as the stable facade. Type-specific
implementation lives under =src/types/=:

#+begin_src text
src/types.ss
src/types/core.ss
src/types/findings.ss
#+end_src

=src/commands/check.ss= remains a command adapter. It asks the types module for
findings and formats the result, but it does not own type facts or type rules.
JSON remains a protocol projection after findings are produced.

** 31.01.3 Environment Facts

=src/types/env.ss= builds a type environment from parser-owned definitions. The
first environment fact records each definition as an =unknown= binding with its
name, kind, path, and selector. This keeps the type pipeline ready for signature
and schema-derived types without making the parser depend on type rules.

The first type rule reports duplicate environment bindings within the same
owner/path as =GERBIL-SCHEME-TYPE-E001=. Names from different source owners stay
distinct.

** 31.01.4 Dispatch Order

The next checker work should flow through the types modules before command or
JSON projection changes:

#+begin_src text
parser facts -> types facts -> types rules -> finding JSON -> check command
#+end_src

Native parser facts are the required test boundary. JSON output is useful for
machines, but it is not the authority for type analysis.
